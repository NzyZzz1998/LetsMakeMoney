# iOS 首次配置输入与工作时间推算设计

## 1. 背景

当前 iOS v0.1 首次配置流程已经能够在 Swift Playgrounds 运行，但设备实测暴露出以下问题：

1. 横屏唤起键盘后，步骤标题和进度条随内容上浮，原始本地化键 `onboarding.step.compensation` 与中文标题重叠。
2. 月薪输入框持续格式化为两位小数，光标默认落在小数部分，不符合用户通常直接输入整数月薪的习惯。
3. 大小周要求填写“双休周周六日期”，用户难以理解锚点日期的含义。
4. 工作时间使用自由文本输入，允许输入无效字符和不完整时间。
5. 工作安排页直接暴露四个时间字段，没有体现“先提供少量信息，再由系统推算”的产品价值。

本设计仅调整首次配置的交互与必要的纯计算辅助逻辑，不改变工资配置文件的现有字段和 schema 版本。

## 2. 目标

- 横屏和竖屏中，标题与进度都保持稳定，不因键盘出现而与内容重叠。
- 月薪支持整数和最多两位小数，但默认按整数输入习惯设计。
- 用“大周/小周”自然语言替代底层周六锚点日期输入。
- 用系统时间选择组件替代自由文本时间输入。
- 根据上班时间、午休时长和默认八小时有效工作时长即时生成完整安排。
- 允许用户继续微调生成结果，并可靠计算有效工时。
- 保持现有 `SalaryConfiguration` JSON 字段、持久化和旧配置兼容。

## 3. 非目标

- 不修改 Settings 页面，本轮仅处理 Onboarding。
- 不新增加班、弹性工作制、跨午夜班次或多段午休。
- 不修改工资计算公式、节假日数据集或月工作日算法。
- 不升级配置 schema，也不新增跨终端同步字段。
- 不重新设计整个 iOS 视觉主题。

## 4. 方案选择

采用“同页渐进式智能推算”。

未采用的方案：

- 四个时间字段始终全部展示：实现简单，但仍像传统表单，不能降低用户理解成本。
- 拆成更多独立步骤：每一步更简单，但会延长首次配置流程。

选定方案在第二页先展示上班时间和午休时长，随后在同一页即时展示推算结果。用户无需额外点击“生成安排”，也无需进入新的向导步骤。

## 5. 页面结构

### 5.1 稳定布局

Onboarding 页面分为三个区域：

1. 固定标题与进度区。
2. 可滚动内容区。
3. 底部操作区。

标题与进度区不进入随键盘压缩的内容布局。键盘出现时，只有内容区调整可见范围并滚动到当前输入框；底部操作区跟随键盘安全区，但不得覆盖输入框。

移除步骤英文标题和动态本地化键文本。页面只显示中文主标题以及无文字标签的进度条。步骤信息通过进度值和页面内容表达。

### 5.2 第一步：收入与休息制度

内容顺序：

1. 月薪。
2. 币种。
3. 休息制度。
4. 仅在选择大小周时出现的“本周是大周还是小周”。

#### 月薪输入

- 配置初值仍为 `0.00`。
- 首次启动时，输入框展示 `0.00`。
- 当值为零且输入框第一次获得焦点时，清空可见文本。
- 已有非零配置进入重新配置时，不自动清空原值；获得焦点时保留原值，便于审阅和修改。
- 使用 decimalPad，允许整数或最多两位小数。
- 输入过程中不插入千位分隔符，避免光标跳动。
- 失去焦点或进入下一步时，将合法值规范化为最多两位小数的展示形式。
- `10000`、`10000.5`、`10000.50` 分别保存为 `1_000_000`、`1_000_050`、`1_000_050` 分。
- 空字符串、仅小数点、超过两位小数或无法解析的文本不能进入下一步。
- 失败时保留用户输入，不回写为零。

金额输入应由一个独立的状态对象或小型组件管理“编辑文本”和“已提交金额”，避免直接通过格式化 Binding 在每次按键后重写字符串。

#### 大小周输入

底层 `alternatingAnchor` 的语义保持不变：它表示一个休息周的周六。

界面定义：

- 大周：本周周六上班。
- 小周：本周周六休息。

锚点计算使用当前设备日历、当前时区和当前日期：

- 用户选择小周：`alternatingAnchor` 写入本周周六。
- 用户选择大周：`alternatingAnchor` 写入下周周六。

“本周周六”以当前自然周为准。周日到周五选择时取即将到来的周六；周六当天选择时取当天。输出仍使用现有 `yyyy-MM-dd` 字符串格式。

重新配置时，根据已保存锚点与当前周周六的周差反推出本周大周或小周。反推失败时不猜测，要求用户重新选择。

### 5.3 第二步：工作安排

#### 基础输入

- 上班时间：系统时间选择组件，默认 `08:00`。
- 午休时长：30 分钟步进，范围 `0` 到 `3` 小时，默认 `2` 小时。
- 午休开始默认 `12:00`。
- 初次推算的目标有效工作时长为 `8` 小时。

时间选择器只能产生合法的小时和分钟，不接受自由文本、英文字符或手写冒号。

#### 初次推算

默认结果为：

```text
上班        08:00
午休        12:00 - 14:00
下班        18:00
有效工时    8 小时
```

推算公式：

```text
午休结束 = 午休开始 + 午休时长
下班时间 = 上班时间 + 8 小时有效工作 + 午休时长
```

v0.1 仅支持同一天内且顺序为“上班 < 午休开始 <= 午休结束 <= 下班”的安排。午休时长为零时，午休开始和午休结束相同，界面显示“无午休”，计算时不扣除午休。

#### 详细调整

推算结果在同一页展示，并允许用户调整：

- 上班时间。
- 午休开始时间。
- 午休结束时间。
- 下班时间。

联动规则：

- 修改午休开始：保持午休总时长不变，自动更新午休结束。
- 修改午休结束：保持午休总时长不变，自动反推午休开始。
- 修改午休时长：保持午休开始不变，更新午休结束。
- 修改上班或下班时间：不再强制保持八小时，按当前四个时间重新计算有效工时。
- 修改午休开始或结束后，也按最终四个时间重新计算有效工时。

自动联动产生越界时间时，不提交无效结果。界面保留最后一组有效配置，并在被修改的字段附近显示可读错误。

#### 有效工时

有效工时继续由 `WorkScheduleMetrics.effectiveWorkSeconds` 计算并写入 `standardWorkSeconds`。为了支持零午休，需要将时间校验从严格的 `lunchStart < lunchEnd` 调整为 `lunchStart <= lunchEnd`，同时保持其他顺序约束。

## 6. 组件与职责

### 6.1 OnboardingView

负责页面组合、步骤切换、键盘布局、校验反馈和保存动作。它不直接实现日期奇偶推算、金额解析或时间运算。

### 6.2 SalaryInputState

负责：

- 编辑文本。
- 零值首次聚焦清空。
- 最多两位小数校验。
- 文本与 minor unit 的安全转换。
- 失焦规范化。

### 6.3 AlternatingWeekResolver

负责：

- 根据当前日期取得本周周六。
- 将大周/小周选择转换成 `alternatingAnchor`。
- 从已有 `alternatingAnchor` 反推当前周类型。

该逻辑属于纯计算，应放在 SalaryCore 中并提供确定时区与日历的测试入口。

### 6.4 WorkScheduleDraft

负责第二页临时状态和联动：

- 上班、午休开始、午休结束、下班时间。
- 午休时长。
- 初次八小时推算。
- 午休开始/结束双向联动。
- 转换为现有 `HH:mm` 配置字段。

SwiftUI 的时间选择器绑定 `Date` 或分钟值，持久化边界再转换成 `HH:mm`，不让自由文本进入配置模型。

## 7. 数据兼容

- 不改变 `SalaryConfiguration` 字段。
- 不改变 `schemaVersion = 1`。
- `monthlySalaryMinor` 继续以分为单位。
- `alternatingAnchor` 继续保存休息周周六的 `yyyy-MM-dd`。
- `workStart`、`lunchStart`、`lunchEnd`、`workEnd` 继续保存 `HH:mm`。
- `standardWorkSeconds` 继续保存最终有效工时。
- 旧配置可直接加载；仅界面将锚点日期解释为当前周类型。

## 8. 错误处理

- 金额无法解析：保留输入，标记月薪字段，禁止下一步。
- 大小周无法生成或反推锚点：要求重新选择本周类型。
- 时间顺序无效：保留最后有效安排，显示字段级错误，禁止下一步。
- 推算超过当天 `23:59`：提示当前安排不受 v0.1 支持，不自动跨日。
- 保存失败：保持现有 Onboarding 完成失败行为，保留草稿并允许重试。

## 9. 可访问性与本地化

- 不再把动态拼接的本地化键作为步骤标题显示。
- 新增大周、小周、午休时长、无午休、时间错误等简体中文文案。
- 时间控件提供明确 accessibility label 和当前值。
- 错误提示与对应字段关联，不能只依赖颜色。
- 横屏、竖屏和键盘状态下，焦点字段始终可见。

## 10. 测试与验收

### 10.1 SalaryCore 单元测试

- 小周生成本周休息周六锚点。
- 大周生成下周休息周六锚点。
- 周六当天、跨月和跨年锚点正确。
- 锚点可反推当前周大周或小周。
- 默认安排推算为 `08:00 / 12:00-14:00 / 18:00 / 28_800 秒`。
- 午休开始与结束双向联动保持固定时长。
- 上下班调整后有效工时重新计算。
- 零午休合法且不扣减工时。
- 非法顺序和跨日安排失败且不覆盖最后有效值。

### 10.2 UI/契约测试

- 步骤标题不包含 `onboarding.step.*` 原始键。
- 月薪默认 `0.00`，零值首次聚焦后为空。
- 月薪接受整数和两位小数，拒绝第三位小数及无效字符。
- 大小周不显示日期文本框。
- 第二页不包含自由文本时间输入框。
- 时间和午休时长变化触发对应联动。
- 返回、取消、关闭、下一步和完成链路保持可用。

### 10.3 iPad 人工验证

- 竖屏和横屏分别打开第一、第二页。
- 横屏聚焦月薪并唤起键盘，确认标题、进度条和中文内容不重叠。
- 输入整数、带一位小数和带两位小数的月薪。
- 选择大小周并到确认页核对月工作日预览。
- 使用专用时间组件调整四个时间。
- 验证午休开始、午休结束和午休时长的双向联动。
- 完成配置，重新打开设置，确认持久化值一致。

## 11. 完成标准

- 用户提出的五项问题均有对应实现与测试证据。
- 横屏键盘不再造成标题或进度重叠。
- 原始本地化键不再出现在 UI。
- 金额输入符合整数优先、最多两位小数的习惯。
- 大小周不要求理解或填写日期。
- 第二页通过合法时间组件完成即时推算与微调。
- 原有配置 schema、保存链路和工资计算回归测试通过。
