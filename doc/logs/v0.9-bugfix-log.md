- Zip：`releases/v0.9/LetsMakeMoney-v0.9-beta-windows-x86_64.zip`
- Zip SHA256：`72972CB344FEE25549DC930A45809C2DAE82F29F9DA4915CE341B995A956E83F`
- 独立解压 EXE SHA256：`BA41DBB61BCE35926C0FB0677CA8722A9F9F71BC8A815A59FE0C82EAC0AC8403`

### 复现步骤

1. 结束旧 `LetsMakeMoney.exe` 进程。
2. 将锁定 Zip 解压到全新目录。
3. 从解压目录启动 `LetsMakeMoney.exe`。
4. 读取 `%APPDATA%\LetsMakeMoney\debug.log` 的本次启动段。

### 实际结果

运行时 validator 能读取两套 `pet-package.json`，但找不到 manifest 列出的原始图集和说明文件：

```text
PetManager.package rejected root=res://assets/pets/packages/letsmakemoney-classic-pro errors=["missing package file: spritesheet.webp", "missing package file: extra-actions.webp", "missing package file: LICENSE.md", "missing package file: SOURCE.md"]
PetManager.package rejected root=res://assets/pets/packages/duoduo-cat errors=["missing package file: spritesheet.webp", "missing package file: extra-actions.webp", "missing package file: LICENSE.md", "missing package file: SOURCE.md"]
PetManager._ready: scanned pets=3
PetManager._ready: current_pet=cat_orange_v2
```

应用随后回退到旧 `cat_orange_v2`，因此候选包中的 v0.9 Classic/多多功能实际不可用。

### 门禁漏检

定向执行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify_v09_package.ps1 `
  -PackagePath .\releases\v0.9\LetsMakeMoney-v0.9-beta-windows-x86_64.zip `
  -SmokeSeconds 3
```

脚本返回退出码 `0` 并打印 `Package verification passed: 0.9-beta`，但同一次隔离启动日志包含上述 `package rejected`。当前 smoke 只检查进程没有退出，没有核对目标宠物包是否成功加载。

### 根因边界

- 已确认：源码目录包含两套 manifest 所列的 `.webp`、`LICENSE.md` 和 `SOURCE.md`。
- 已确认：导出配置没有显式 include filter；运行时按文件路径和 SHA256 读取原始载荷。
- 已确认：源码态宠物包测试通过，但导出包运行态失败。
- 高度可能：Godot 导出只保留了可导入资源的重映射结果，运行时按原始文件路径读取与哈希校验的合同没有对应的导出载荷策略。
- 待修复阶段确认：应显式打包原始载荷，还是将 importer/validator 改为消费 Godot 导出资源；不得在未验证哈希、许可和回退合同前任选其一。

### 根因与修复

- Godot 默认把 `.webp` 导入为纹理资源，导出时保留重映射后的纹理数据；运行时宠物包合同则需要原始 WebP 字节执行 SHA256 校验和动态图集解码。
- 四个宠物包图集改为 Godot 官方 `Keep File (exported as is)` 导入模式，原始字节随项目导出。
- 保留 manifest、文件 SHA256、许可、来源和目录边界校验，没有跳过或弱化 validator。
- 包验证器现在读取隔离运行日志，必须同时出现 Classic 与多多的 `shadow_loaded`；任一包出现 `package rejected` 即失败。
- 新增 `test_v09_exported_pet_payload.ps1`，能够拒绝旧坏包并验证当前导出合同。

### 最小修复与复验门禁

1. 修复导出运行时载荷，使 Classic 与多多在独立解压包中均记录 `shadow_loaded`。
2. 保持源码包 schema、文件哈希、许可和来源校验，不得通过跳过文件检查来掩盖问题。
3. 扩展 `verify_v09_package.ps1`：解析隔离 `debug.log`，任一目标包 `rejected`、缺少两个 `shadow_loaded` 或回退到错误默认宠物时返回非零。
4. 新增能够先失败、修复后通过的导出包回归测试。
5. 重新生成 Zip、EXE、DLL 哈希；旧候选作废。
6. 从候选身份锁定开始重新执行完整独立验收。
