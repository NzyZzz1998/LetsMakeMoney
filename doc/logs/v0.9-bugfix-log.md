## V09-BUG-001 导出候选无法加载 Classic Pro 与多多

| 项目 | 内容 |
|---|---|

| 严重度 | 发布阻塞 |

| 影响范围 | Classic Pro、多多、宠物选择、动画状态机真实观感、动态命中区、回滚与发布门禁 |

### 复现对象

- Zip：`releases/v0.9/LetsMakeMoney-v0.9-beta-windows-x86_64.zip`
- Zip SHA256：`72972CB344FEE25549DC930A45809C2DAE82F29F9DA4915CE341B995A956E83F`

### 复现步骤

1. 结束旧 `LetsMakeMoney.exe` 进程。
2. 将锁定 Zip 解压到全新目录。
3. 从解压目录启动 `LetsMakeMoney.exe`。
4. 读取 `%APPDATA%\LetsMakeMoney\debug.log` 的本次启动段。

### 实际结果

运行时 validator 能读取两套 `pet-package.json`，但找不到 manifest 列出的原始图集和说明文件：

```text
PetManager.package rejected root=res://assets/pets/packages/letsmakemoney-classic-pro errors=["missing package file: spritesheet.webp", "missing package file: extra-actions.webp", "missing package file: LICENSE.md", "missing package file: SOURCE.md"]
PetManager.package rejected root=res://assets/pets/packages/duoduo-cat errors=["missing package file: spritesheet.webp", "missing package file: extra-actions.webp", "missing package file: LICENSE.md", "missing package file: SOURCE.md"]
PetManager._ready: scanned pets=3
PetManager._ready: current_pet=cat_orange_v2
```

应用随后回退到旧 `cat_orange_v2`，因此候选包中的 v0.9 Classic/多多功能实际不可用。

### 门禁漏检

定向执行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify_v09_package.ps1 `
  -PackagePath .\releases\v0.9\LetsMakeMoney-v0.9-beta-windows-x86_64.zip `
  -SmokeSeconds 3
```

脚本返回退出码 `0` 并打印 `Package verification passed: 0.9-beta`，但同一次隔离启动日志包含上述 `package rejected`。当前 smoke 只检查进程没有退出，没有核对目标宠物包是否成功加载。

### 根因边界

- 已确认：源码目录包含两套 manifest 所列的 `.webp`、`LICENSE.md` 和 `SOURCE.md`。
- 已确认：导出配置没有显式 include filter；运行时按文件路径和 SHA256 读取原始载荷。
- 已确认：源码态宠物包测试通过，但导出包运行态失败。
- 高度可能：Godot 导出只保留了可导入资源的重映射结果，运行时按原始文件路径读取与哈希校验的合同没有对应的导出载荷策略。
- 待修复阶段确认：应显式打包原始载荷，还是将 importer/validator 改为消费 Godot 导出资源；不得在未验证哈希、许可和回退合同前任选其一。

### 根因与修复

- Godot 默认把 `.webp` 导入为纹理资源，导出时保留重映射后的纹理数据；运行时宠物包合同则需要原始 WebP 字节执行 SHA256 校验和动态图集解码。
- 四个宠物包图集改为 Godot 官方 `Keep File (exported as is)` 导入模式，原始字节随项目导出。
- 保留 manifest、文件 SHA256、许可、来源和目录边界校验，没有跳过或弱化 validator。
- 包验证器现在读取隔离运行日志，必须同时出现 Classic 与多多的 `shadow_loaded`；任一包出现 `package rejected` 即失败。
- 新增 `test_v09_exported_pet_payload.ps1`，能够拒绝旧坏包并验证当前导出合同。

### 定向复验结果

- `package_v09.ps1`：通过。
- `verify_v09_package.ps1`：通过。
- `test_v09_exported_pet_payload.ps1`：通过。
- 新 Zip SHA256：`7D1D7199362779AE2EE4E5DC9F8278C0B730D26AD39DB761A9680930E0AFD4BD`。
- 新 EXE SHA256：`B6CACC6D802E85DE6FBEB0D8BD0C5864B3ADEC2E72E9EDF72F0B6BD88D1D1E07`。
- Native DLL SHA256：`E3E2030003A7DA725446A3873C3EC2E19D9442B98A67F24A771E76BD0BAD5089`。
- 结论：导出阻塞关闭；因动画交互合同随后修订，仍需生成最终候选并从头执行独立验收。

### 最小修复与复验门禁

1. 修复导出运行时载荷，使 Classic 与多多在独立解压包中均记录 `shadow_loaded`。

## V09-BUG-002 状态感知单击无法进入动画控制器

| 项目 | 内容 |

| 发现日期 | 2026-07-22 |
| 严重度 | 发布阻塞 |
| 状态 | 已修复，定向复验通过 |
| 发现阶段 | S5.5 独立 `/acceptance` |
| 影响范围 | Classic 与多多的状态感知单击、动作恢复、动画语义日志及发布签核 |

### 复现对象

- Zip：`releases/v0.9/LetsMakeMoney-v0.9-beta-windows-x86_64.zip`
- Zip SHA256：`36CEE0D4C73CDBBA876F59BA84C259C04F3E2F95D959EF147135009985A84465`
- EXE SHA256：`B30E5E4409B8411738ABDE84AD0EC52E92DF3F975A9D69EFA1DB6CE6E5DD2FA1`
- Native DLL SHA256：`E3E2030003A7DA725446A3873C3EC2E19D9442B98A67F24A771E76BD0BAD5089`

### 实际结果

真实 GUI 中，单击能被稳定分类为 `clicked_single`，但画面保持基础 `working_loop`，日志没有 `pet.animation.requested`、`pet.animation.started` 或 `pet.animation.finished`。真实 Pet 场景探针进一步确认，交互状态进入单击后动作令牌仍为 `0`，最终也没有恢复基础交互状态。

### 根因

`src/scenes/pet/pet.gd:146` 使用严格类型变量接收未类型化数组：

```gdscript
var candidates: Array[String] = ActionProfileScript.interaction_candidates(...)
```

运行时报告 `Trying to assign an array of type "Array" to a variable of type "Array[String]"`。错误发生在动画控制器请求之前，导致动作候选无法提交，PetManager 的交互状态停留在 `CLICKED_SINGLE`。

### 自动测试缺口

当前测试已覆盖宠物包动作存在性、逐帧时长、动作 Profile 和动画控制器，但没有实例化真实 Pet 场景并覆盖 `PetManager.state_changed -> _request_interaction_action -> animation controller` 完整集成路径。因此自动门禁通过不能证明产品场景能实际播放状态感知动作。

### 最小修复方向

1. 统一动作候选 API 与调用端的类型合同，保留错误处理，不把所有失败伪装为成功。
2. 检查同类基础状态、业务事件、环境动作和跑动候选调用是否使用相同类型边界，但只修改有代码证据的路径。
3. 新增真实 Pet 场景集成测试，至少断言请求、开始、完成、动作令牌释放和基础状态恢复。
4. 重新运行宠物包、动画、行为基线及 v0.9 全量回归。
5. 重新导出、打包并记录新 Zip/EXE SHA256，旧候选作废。
6. 用新解压候选定向复验工作、清醒休息和睡眠下的单击动作，再继续剩余独立验收。

### 证据

- `.tmp_acceptance/v0.9-s55-20260722-163359/evidence/gui-after-single-clicks.png`
- `.tmp_acceptance/v0.9-s55-20260722-163359/evidence/gui-after-run-left.png`

### 修复与定向复验

- `src/utils/pet_action_profile.gd` 通过 `_with_base_candidates()` 统一返回 `Array[String]`，消除数组拼接产生未类型化 `Array` 的运行时边界；相同合同同时覆盖单击、跑动、环境动作和业务事件候选。
- `scripts/verify_v09_pet_integration.gd` 新增真实 Pet 场景集成测试，覆盖 `PetManager -> pet.gd -> action profile -> animation controller` 完整链路。该测试在修复前稳定复现原错误，修复后通过。

## V09-BUG-003 低比例缩放时 Panel 内容脱离外壳

| 项目 | 内容 |
|---|---|
| 发现日期 | 2026-07-22 |
| 严重度 | 发布阻塞 |
| 状态 | 已修复，定向复验通过 |
| 发现阶段 | S5.5 独立 `/acceptance` |
| 影响范围 | Panel 折叠/展开布局、缩放设置、交互命中区和桌宠邻接观感 |

### 实际结果

显示缩放设为 58% 后，Panel 白色外壳缩小，状态文字、进度条和下一节点仍按较大宽度排列，内容横向脱离外壳并进入桌宠区域。调试黑底只帮助观察窗口范围，不是该布局缺陷的原因。

### 根因证据

- `src/scenes/panel/panel.gd:set_display_scale()` 会缩放 Panel 外壳和内容容器。
- `src/scenes/panel/panel.tscn` 的折叠态进度条保留固定 `268px` 最小宽度。
- `_apply_collapsed_text_style()` 只缩放 `collapsed_content`、字体和间距，没有同步更新进度条宽度。
- 子节点固定最小宽度大于缩小后的外壳宽度，导致容器布局溢出。

### 最小修复方向

1. 统一折叠态和展开态所有固定尺寸的缩放入口，避免场景静态尺寸与运行时尺寸并存。
2. 核对 Header、Footer、ProgressBar、MetricsRow、Separator 和 ScheduleRow 的最小尺寸及文字裁切策略。
3. 同步核对 Panel 视觉尺寸、交互矩形和主窗口几何，避免修复显示后命中区仍使用旧尺寸。
4. 新增 50%、58%、100%、150% 的确定性布局测试，覆盖折叠/展开、长金额和最长状态文案。
5. 用新候选真实截图复验，不用位图缩放代替布局验证。

### 证据

- `.tmp_acceptance/v0.9-s55-20260722-163359/evidence/gui-scale-058-layout-failure.png`

### 修复与定向复验

- `src/scenes/panel/panel.gd` 将折叠进度条、展开指标行、指标分隔线和主分隔线纳入统一 `_display_scale` 合同。
- `scripts/verify_v09_window_experience.gd` 实例化真实 Panel，覆盖 50%、58%、100%、150% 的固定子节点尺寸与外壳约束。该测试在修复前于 50%、58%、150% 失败，修复后全部通过。
- 新候选在隔离配置中以 58% 缩放真实启动，折叠 Panel 的文字、进度条与外壳保持同一范围。
- 截图：`.tmp_acceptance/v0.9-bugfix-20260722/evidence/panel-scale-058-after-fix.png`。
