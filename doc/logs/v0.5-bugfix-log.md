# LetsMakeMoney v0.5 Bugfix Log

This file records v0.5 acceptance defects and follow-up bugfix candidates.
It intentionally keeps bugfix details out of `doc/releases/v0.5/progress_v0.5.md`.

## 2026-07-09 Acceptance Findings

### V05-BUG-001 Settings save failure feedback can report success while config is not updated

- Status: Fixed and targeted acceptance passed
- Source: v0.5 beta acceptance run against `releases/v0.5/LetsMakeMoney-v0.5-beta-windows-x86_64/`.
- Evidence: Settings save success and no-change paths were visible and worked. During a simulated failure attempt, the UI still showed the save-success message, while `%APPDATA%\LetsMakeMoney\config.json` remained at the previous `monthly_salary` value before the later Wizard completion.
- Impact: Save failure feedback cannot be accepted as fully reliable. This does not block the normal save/no-change path, but it weakens the failure-path acceptance.
- Suggested fix: Review `Config.save` return/error propagation and Settings feedback refresh timing. Add a focused verification path that can deterministically force save failure without mutating user ACLs.
- Fix summary: `Config.save()` now returns `bool`, records `config_save_success` / `config_save_failed`, verifies the persisted file content, and exposes `get_last_save_error()`. Settings now snapshots config before applying form values, rolls back memory config on save failure, keeps user input visible, shows failure feedback, and logs `settings_save_success`, `settings_save_no_change`, or `settings_save_failed`.
- Targeted acceptance: Passed on 2026-07-09. Success path changed `monthly_salary` from `1000` to `1100` and logged `config_save_success` / `settings_save_success`; no-change save logged `settings_save_no_change`; forced ACL-denied save showed failure feedback, kept the unsaved `1200` input visible, preserved config at `1100`, and logged `config_save_failed` / `settings_save_failed` with the open-file error.

### V05-BUG-002 debug.log lacks fine-grained Settings/Wizard semantic events

- Status: Fixed and targeted acceptance passed
- Source: v0.5 beta acceptance log review.
- Evidence: `debug.log` records startup, native bridge health, tray setup, passthrough suspend/resume, modal open/close, and window setup events. It does not clearly record Settings save result, Wizard step transitions, or Wizard completion as stable semantic events.
- Impact: Core runtime logging is usable, but `V05` log acceptance is only partial for Settings/Wizard workflows.
- Suggested fix: Add low-noise events such as `settings_save_result`, `wizard_opened`, `wizard_step_changed`, and `wizard_finished`.
- Fix summary: Added Settings save semantic events and Wizard events for opened, step changed, finished, finish failed, cancelled, and closed.
- Targeted acceptance: Passed on 2026-07-09. Logs include `wizard_opened`, `wizard_step_changed: from=1 to=2`, `from=2 to=3`, `from=3 to=4`, `wizard_finished`, `wizard_closed: reason=finished`, `wizard_cancelled`, and `wizard_closed: reason=cancelled`.

### V05-BUG-003 Tray left-click restore still requires manual taskbar verification

- Status: Fixed in code, pending native package retest
- Source: v0.5 beta acceptance run with Computer Use.
- Evidence: The app menu item for hiding to tray hides the window and leaves the process running. Computer Use cannot reliably enumerate or click the Windows notification-area tray icon, so the tray left-click restore path could not be completed by automation in this run.
- Impact: Hide-to-tray and native tray setup are evidenced, but left-click restore remains a human acceptance step.
- Suggested fix: Keep this in the manual release checklist, or add a debug-only native tray command hook for automated acceptance.
- Manual retest result: The user confirmed the correct product expectation: tray icon left-click should hide/show the window in both pure-pet and non-pure-pet modes. The difference is only the taskbar policy after restore: non-pure-pet shows both the desktop pet and a taskbar entry; pure-pet restores the desktop pet without a taskbar entry.
- Fix summary: Native tray icon left-click now emits a distinct `COMMAND_LEFT_TOGGLE = 5`, while the tray menu show/hide item keeps the existing `COMMAND_TOGGLE = 1`. `Main._on_tray_left_toggle_requested()` now toggles visibility for both modes and re-applies the window policy after show, so pure-pet mode restores without exposing the taskbar entry.
- Follow-up finding: Manual retest showed the window restored in pure-pet mode, but the taskbar entry still appeared. Root cause: `Main` invalidated its taskbar cache, but `WindowsPlatform` also cached taskbar visibility per window and skipped the native `set_taskbar_visible(false)` call after native `ShowWindow`. Fix: invalidate the WindowsPlatform taskbar cache after `setup_window` and after native show, and log every real native `WindowsPlatform.set_taskbar_visible` call.
