## 1. 项目概述

### 1.1 产品定位

**LetsMakeMoney（赚钱模拟器）** 是一款桌面宠物软件，通过可爱的小动物形象陪伴用户工作，并基于用户设定的薪资参数实时计算"这一刻已经赚了多少钱"。

产品兼顾两种价值：
- **情感陪伴**：扁平卡通风格的小动物在桌面上敲键盘、休息、互动，提供治愈感
- **实用反馈**：严肃真实的薪资计算逻辑，让用户时刻感知时间价值

### 1.2 目标用户

- 长期在电脑前工作的上班族、自由职业者
- 希望通过趣味化方式感知时间价值的人群
- 喜欢桌面宠物/治愈系软件的用户

### 1.3 核心价值

> 用一只努力敲键盘的小动物告诉你："你正在赚钱"

#### 3.1.3 薪资计算规则

**输入参数**（用户设定）：
- 月薪 `monthly_salary`
- 休息模式 `rest_mode`：`single`（单休，仅周日休息）或 `double`（双休，周六日休息）
- 每日工作小时数 `work_hours_per_day`
- 上班时间 `work_start_time`（如 09:00）
- 下班时间 `work_end_time`（如 18:00）

**每月工作天数**：由系统根据休息模式 + 当前月份日历自动计算，无需用户手动输入。

| 模式 | 工作天数 | 示例（2026年6月） |
|------|---------|-------------------|
| 双休 | 当月周一至周五的天数 | 22 天 |
| 单休 | 当月周一至周六的天数 | 26 天 |

每月首日自动重新计算当月工作天数，薪资引擎使用新值更新 `rate_per_second` 和 `hourly_rate`。

**派生值**：
- 当月工作天数 `work_days = calc_work_days(year, month, rest_mode)` — 根据日历经月份自动计算
- 每秒收入 `rate_per_second = monthly_salary / (work_days × work_hours_per_day × 3600)`
- 时薪 `hourly_rate = monthly_salary / (work_days × work_hours_per_day)`
- 每月首日自动重算 `work_days`，连带更新 `rate_per_second` 和 `hourly_rate`

**实时计算**：
- **工作时间判定**：当前时间在 `work_start_time` 与 `work_end_time` 之间 → 角色处于 WORKING 状态，数字增长
- **今日已赚**：
  - 上班前（当前时间 < `work_start_time`）：0
  - 工作中（`work_start_time` ≤ 当前时间 ≤ `work_end_time`）：`(当前时间 - work_start_time) × rate_per_second`
  - 下班后（当前时间 > `work_end_time`）：`work_hours_per_day × 3600 × rate_per_second`（封顶为全天工作总收入）
- **本月累计**：`monthly_salary × (今日日期 / 当月天数)`
- **工作进度**：`今日工作已过秒数 / (work_hours_per_day × 3600)`，下班后为 100%

**非工作时间**：
- 角色切换到 RESTING 状态（睡觉/打盹动画）
- "今日已赚"在上班前显示 0，下班后保持全天工作总收入值（不再增长）

### 3.2 角色与视觉

#### 3.2.1 视觉风格

- **风格**：扁平卡通矢量
- **v0.1 占位方案**：使用免费像素风素材（如 [itch.io Cat 2D Pixel Art](https://xzany.itch.io/cat-2d-pixel-art)）作为占位素材进行原型开发。v1.0 正式版替换为扁平卡通矢量素材。
- **线条**：清晰、柔和
- **色彩**：鲜明温暖
- **尺寸**：原生像素 64×64 px（v0.1 占位素材），显示时缩放到约 120×120 px，用户可调（50% - 200%）

#### 3.2.2 角色规划

**v0.1 初始角色（2-3 个）**：
- 小猫 🐱
- 小狗 🐶
- 仓鼠 🐹（可选，视开发进度）

**架构要求**：角色通过 `PetResource` (.tres) 资源文件定义，新增角色只需在 `pets/` 目录添加资源文件，无需修改核心代码。

**v0.1 实际交付**：当前仅完成 `cat` 一套 PetResource / SpriteFrames 链路；小狗、仓鼠未创建。可运行素材仍为兔子 sprite 占位，app icon 已替换为用户确认过的小猫图标。正式小猫动画质量未达到直接接入标准，作为 v0.2 素材 Spike 或后续版本继续探索。

#### 3.2.3 动画状态机

状态切换由 Pet Manager 根据薪资引擎的工作时间判定和鼠标事件驱动。

```
IDLE       待机/呼吸动画（鼠标未靠近）
WORKING    疯狂敲键盘（工作时间内）
RESTING    睡觉/打盹（非工作时间）
HOVER      看向鼠标（鼠标悬停时）
CLICKED    被点击反馈动画（单击/双击/长按各有变体）
```

##### 3.2.3.1 动画规格

| 状态 | 帧数 | 循环/单次 | 周期 | 说明 |
|------|------|-----------|------|------|
| IDLE | 4 | 循环 | 2000ms | 站立呼吸/微微摆动，鼠标未靠近时默认播放 |
| WORKING | 4 | 循环 | 800ms | 快速敲键盘动作，工作时间播放 |
| RESTING | 2 | 循环 | 3000ms | 闭眼起伏呼吸/打盹，非工作时间播放 |
| HOVER | 1 | 静态 | — | 切换到"看向鼠标"的单帧姿态，鼠标离开后回到 IDLE |
| CLICKED（单击）| 3 | 单次 | 600ms | 被摸头/弹跳反馈，播放完后回到先前状态 |
| CLICKED（双击）| 3 | 单次 | 800ms | 欢呼/撒花变体，播放完后回到先前状态 |
| CLICKED（长按）| 2 | 循环 | 1000ms | 呆住/疑惑表情，松开后回到先前状态 |

**帧来源标注**：v0.1 占位素材的 IDLE / RESTING / CLICKED 帧来自免费像素素材包；WORKING（敲键盘）和 HOVER 帧需手动调整或创建。

**v0.1 实际交付**：状态机与动画名称链路已打通，但 WORKING / HOVER 仍以现有占位帧复用方式实现，不代表最终动作表现。v0.2 不把正式动画接入作为阻塞项，除非素材 Spike 产出质量稳定。

##### 3.2.3.2 Godot 动画实现

使用 AnimatedSprite2D + SpriteFrames 资源。每个角色一个 `.tres` SpriteFrames 文件，包含上述所有动画。PetResource 存储对 SpriteFrames 资源的引用。

```
pets/
  cat/
    cat_sprite_frames.tres   # SpriteFrames 资源
    cat_resource.tres         # PetResource（引用 sprite_frames）
  dog/
    dog_sprite_frames.tres
    dog_resource.tres
  hamster/
    hamster_sprite_frames.tres
    hamster_resource.tres
```

### 3.3 UI 设计

| 模式 | 行为 |

| 融入桌面 | 像桌面小部件，点击桌面空白处可见 |

**两种模式统一行为**：

| 行为 | 说明 |
|------|------|

| Alt+Tab 列表 | 不出现 |

**v0.1 实际偏差**：
- 当前 WindowsPlatform 为调试稳定性设置：`borderless=false`、`transparent_bg=false`、`always_on_top=false`、`size=900×500`。
- v0.1 仍显示在任务栏/Alt+Tab 中，便于找回窗口；“不显示任务栏/Alt+Tab”未实现。

#### 3.3.2 薪资面板

**简洁模式（常态）**：
```
┌──────────────┐
│  💰 ¥12.34   │
└──────────────┘
```
半透明背景，角色旁边显示今日已赚。

**展开模式（鼠标悬停）**：
```
┌────────────────────────┐
│  今日已赚    ¥12.34     │
│  本月累计    ¥370.20    │
│  时薪        ¥46.88     │
│  ───────────────────    │
│  工作进度    2h/8h  ██░░ │
│  状态        🐱 努力工作中 │
└────────────────────────┘
```

**隐私控制**：展开模式的每一项内容支持用户在设置中开启/关闭，默认全显示。用户可隐藏敏感项（如本月累计、时薪）。

**交互时序**：
- 悬停面板 300ms → 触发平滑展开（ease-in-out，200ms 动画）
- 鼠标离开面板 500ms → 触发平滑收起（ease-in-out，200ms 动画）
- 期间鼠标重新进入 → 取消收起，保持展开

**面板定位**：默认在角色右侧。若角色贴屏幕右边缘（右侧空间不足），面板自动出现于左侧；若贴底部则上方弹出。每次展开时动态计算最佳位置。

#### 3.3.3 位置与记忆

- 首次启动默认位置：屏幕右下角
- 用户拖拽后，退出时保存屏幕坐标
- 下次启动还原上次位置
- 若保存的位置在当前屏幕外（多显示器切换）， fallback 到右下角

#### 3.3.4 透明度与大小

| 项 | 范围 | 默认 |
|----|------|------|
| 角色透明度 | 20% - 100% | 100% |
| 角色缩放 | 50% - 200% | 100% |

### 3.6 架构设计

#### 3.6.1 模块结构

```
┌─────────────────────────────────────────────────────┐
│                   Main App (Node)                    │
│          桌面窗口管理、置顶/嵌入模式、托盘系统            │
├─────────┬─────────┬──────────┬──────────┬────────────┤
│ Salary  │ Pet     │ Panel    │ Settings │ DragResize │
│ Engine  │ Manager │  System  │ System   │  System    │
│ (Auto)  │ (Auto)  │ (Auto)   │ (Auto)   │  (Auto)    │
├─────────┴─────────┴──────────┴──────────┴────────────┤
│                    Config (Auto)                      │
│          用户配置持久化（透明度/大小/位置/薪资）            │
└─────────────────────────────────────────────────────┘
```

#### 3.6.2 模块职责

##### Salary Engine（薪资引擎）— Autoload 单例

- 接收薪资参数，计算每秒收入、时薪
- 判定当前是否为工作时间
- 根据休息模式和当前月份日历自动计算当月工作天数
- 每月首日自动刷新工作天数和时薪
- 提供 `get_earnings_today()`、`get_earnings_this_month()`、`get_rate_per_second()`、`is_working_hours()`、`get_work_days_this_month()` 接口

##### Pet Manager（角色管理器）— Autoload 单例

- 加载 `pets/` 目录下的所有 `PetResource`
- 管理角色状态机（IDLE / WORKING / RESTING / HOVER / CLICKED）
- 根据薪资引擎判定切换 WORKING / RESTING
- 提供切换角色接口

##### Panel System（面板系统）— Autoload 单例

- 简洁模式：显示今日已赚数字
- 展开模式：根据用户配置显示对应详情项
- 鼠标悬停触发展开

##### Settings System（设置系统）— 独立 Scene

- 配置项 UI（见 3.4）
- 保存时写入 Config 单例
- 实时应用变更（如调透明度立即生效）

##### Drag & Resize System — Autoload 单例

- 拖拽角色区域移动窗口
- 右键菜单（设置/切换角色/切换模式/退出）
- 位置记忆与还原

##### Config — Autoload 单例

- 加载/保存 `config.json`
- 提供配置读写接口
- 配置变更通知（观察者模式）

#### 3.6.3 跨平台抽象层

Windows 相关逻辑（托盘、窗口置顶/嵌入桌面、开机自启）封装在 `platform/` 目录下，通过接口抽象：

```
platform/
  PlatformInterface.gd    # 抽象接口
  WindowsPlatform.gd      # Windows 实现
  # 未来：MacPlatform.gd、LinuxPlatform.gd
```

v0.1 只实现 WindowsPlatform，但所有调用通过 PlatformInterface，为后续移植铺路。

### 4.3 运行模式

v0.2 将应用拆分为两种运行形态：**桌宠模式**面向日常使用，**Debug 模式**面向开发验证。两者通过同一套主场景运行，但窗口属性、可见调试节点和输入命中策略不同。

| 模式 | 触发条件 | 窗口形态 | 面向对象 | 说明 |
|------|----------|----------|----------|------|
| 桌宠模式 | `debug_mode=false` 或字段缺失 | 无边框、透明背景、置顶悬浮 | 普通用户 | 默认启动形态，隐藏所有 debug UI |
| Debug 模式 | `debug_mode=true` | 普通大窗口、不透明背景 | 开发/测试 | 显示输入区、状态文字和调试日志 |

#### 桌宠模式（默认）

当 `debug_mode=false` 或旧配置文件缺少 `debug_mode` 字段时，应用按桌宠模式启动：

- 窗口无边框，背景透明。
- 默认保留任务栏入口，避免没有托盘时用户无法找回程序。
- DebugInputArea、DebugStatus、调试边框、调试提示默认不可见，且不参与布局占位。
- 小猫和薪资 Panel 保持可见，并根据用户配置应用缩放、透明度、窗口位置。
- 置顶悬浮仍是 v0.2 的主验收模式；“融入桌面”保留为实验能力，不作为核心验收。
- 窗口尺寸应尽量贴合桌宠与 Panel 的实际视觉区域，避免出现明显大块透明可点击矩形。

#### Debug 模式

当 `%APPDATA%/LetsMakeMoney/config.json` 中 `debug_mode=true` 时，应用按调试模式启动：

- 显示较大的普通调试窗口，便于观察 Pet、Panel 和输入区域。
- 显示 debug 输入区和状态文字。
- 输出关键交互日志，包括 click、double click、hold、drag started、drag saved、context menu。
- 用于开发验证，不作为普通用户默认体验。

v0.2 不要求在设置界面提供 `debug_mode` 开关，优先通过配置文件修改，降低 UI 复杂度。

### 4.4 窗口与输入需求

#### 4.4.1 窗口行为

| 场景 | 预期行为 | 失败降级 |
|------|----------|----------|
| 首次启动 | 默认进入桌宠模式，窗口位于屏幕右下角或上次保存位置 | 位置无效时回退到右下角 |
| 置顶悬浮 | 窗口保持在普通应用窗口之上 | 置顶失败时仍保持可见和可拖拽 |
| 透明背景 | 背景透明，仅小猫和 Panel 可见 | 透明失败时显示普通窗口并记录 warning |
| 空白区域点击 | 尽量穿透到桌面或下层窗口，不触发应用交互 | 穿透不可用时收紧窗口尺寸，减少误拦截 |
| 关闭窗口 | 默认隐藏到托盘，不结束进程 | 托盘不可用时允许关闭退出，并记录 warning |
| 多显示器切换 | 保存位置不在任何屏幕内时回退到当前主屏右下角 | 不使用旧坐标 |

空白区域穿透优先使用 Godot 4.7 可用窗口能力；若 Godot API 无法稳定覆盖 Windows 需求，再评估是否在 WindowsPlatform 中补充平台特定实现。无论采用哪种实现，角色区域交互必须保持稳定。

#### 4.4.2 角色输入

v0.2 Beta 的输入判定以“小猫区域”为准。透明窗口中小猫以外的空白区域不应承担交互职责，避免用户点击桌面时被应用误拦截。

| 触发方式 | 判定规则 | 反应 | 备注 |
|----------|----------|------|------|
| 鼠标进入小猫区域 | 命中小猫可交互区域 | PetManager 切到 HOVER | 不要求 Panel 区域触发 Pet hover |
| 鼠标离开小猫区域 | 指针离开可交互区域且未拖拽 | 回到自动状态 | 自动状态由工作/休息时间决定 |
| 单击小猫 | 按下后移动未超过拖拽阈值，且未进入双击窗口 | 播放单击反馈 | 不应和拖拽互相误触 |
| 双击小猫 | 两次单击在双击时间窗口内 | 播放双击反馈 | v0.1 曾出现双击不触发，v0.2 必须回归验证 |
| 长按小猫 | 按下持续超过长按阈值 | 播放长按反馈 | 松开后恢复自动状态 |
| 拖拽小猫 | 按下后屏幕鼠标位移超过阈值 | 移动整个窗口 | 使用屏幕绝对位移，避免窗口移动速度异常 |
| 右键小猫 | 右键按下或释放命中小猫区域 | 弹出角色右键菜单 | 菜单位置靠近鼠标，不能跑出屏幕 |

#### 4.4.3 右键角色菜单

v0.2 保留 v0.1 菜单结构，并把设置、退出、窗口模式作为重点验收路径：

```
├── 设置
├── 切换角色
│     └── 当前角色（仅一个角色时仍可点击，不报错）
├── 窗口模式
│     ├── 置顶悬浮
│     └── 融入桌面（实验）
├── 重新运行向导
├── ──────────
├── 关于 LetsMakeMoney
└── 退出
```

菜单项行为：

- `设置`：打开设置对话框，不重复创建多个设置窗口。
- `切换角色`：v0.2 可以只有当前角色，但菜单不能报错。
- `窗口模式 > 置顶悬浮`：设置 `window_mode=top` 并立即应用。
- `窗口模式 > 融入桌面`：设置 `window_mode=embed`，若平台不支持则降级为非置顶普通窗口。
- `重新运行向导`：打开首次启动向导，用于测试和重配薪资。
- `关于 LetsMakeMoney`：打开关于窗口。
- `退出`：保存配置并结束进程，不只是隐藏窗口。

### 4.5 系统托盘与开机自启

#### 4.5.1 系统托盘

v0.2 Beta 需要把 v0.1 的托盘降级方案升级为正式托盘能力。托盘是桌宠模式的主要驻留入口，用于找回、隐藏和退出应用。

- 应用启动后显示系统托盘图标，图标使用当前 app icon 或其小尺寸变体。
- 左键点击托盘图标切换显示/隐藏桌宠窗口。
- 右键托盘图标显示菜单：显示/隐藏、设置、关于 LetsMakeMoney、退出。
- 点击窗口关闭按钮时优先隐藏到托盘，不直接退出；通过托盘菜单或右键菜单“退出”才真正退出。
- 退出前保存窗口位置和当前配置。
- 若托盘初始化失败，应用应继续运行，并至少保留任务栏入口和角色右键菜单作为降级路径。

**托盘菜单结构：**

```
├── 显示/隐藏
├── 设置
├── 关于 LetsMakeMoney
├── ──────────
└── 退出
```

| 菜单项 | 行为 |
|--------|------|
| 显示/隐藏 | 当前窗口可见时隐藏；隐藏时恢复显示到上次位置 |
| 设置 | 若窗口隐藏，应先显示窗口或直接弹出设置窗口 |
| 关于 LetsMakeMoney | 弹出关于窗口 |
| 退出 | 保存配置，移除临时托盘状态，结束进程 |

#### 4.5.2 开机自启

开机自启作为 v0.2 正式设置项：

- 设置页提供“开机自启”开关，默认关闭。
- 开启时写入 Windows 当前用户自启动配置，关闭时移除对应配置。
- 自启动目标应指向已导出的 `LetsMakeMoney.exe`；开发态运行时若无法确定 exe 路径，应给出可理解提示，不静默失败。
- 自启动只要求 Windows 支持，后续平台通过 PlatformInterface 扩展。

Windows 自启动优先使用当前用户级别配置，不要求管理员权限。具体实现可以选择注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 或启动文件夹快捷方式，但必须通过 PlatformInterface 封装，避免散落在 UI 代码中。

### 4.6 设置体验优化

v0.2 Beta 的设置优化围绕“降低测试和日常使用成本”展开：

#### 4.6.1 设置项明细

| 分类 | 选项 | 控件类型 | 默认值 | v0.2 要求 |
|------|------|----------|--------|-----------|
| 显示 | 角色缩放 | 滑块 + 百分比文本 | 100% | 保存后立即反映到主界面 |
| 显示 | 透明度 | 滑块 + 百分比文本 | 100% | 保存后立即反映到主界面 |
| 显示 | 窗口模式 | 开关/分段控件 | 置顶悬浮 | `top` 为主路径，`embed` 为实验路径 |
| 显示 | 重置窗口位置 | 按钮 | - | 将窗口移动到当前主屏幕右下角并保存 |
| 通用 | 开机自启 | 开关 | 关闭 | 写入/移除 Windows 当前用户自启动配置 |
| 通用 | 关闭时隐藏到托盘 | 开关 | 开启 | 控制关闭按钮是隐藏到托盘还是退出 |

#### 4.6.2 设置保存行为

- 点击“保存”才写入 Config；点击“取消”或关闭设置窗口放弃未保存修改。
- 保存后立即应用缩放、透明度、窗口模式、开机自启、关闭行为。
- 开机自启写入失败时，设置窗口应提示失败原因，并且不把 UI 状态误显示为已开启。
- 恢复默认设置需要二次确认，避免误清用户配置。
- 重置窗口位置不需要二次确认，但应立即移动窗口并保存。
- 设置窗口不应被透明穿透影响，打开时必须可点击、可关闭。

#### 4.6.3 恢复默认设置范围

v0.2 的“恢复默认设置”默认只恢复非敏感显示/运行配置：

| 配置项 | 是否恢复 | 原因 |
|--------|----------|------|
| `scale` | 是 | 显示设置，可安全恢复 |
| `opacity` | 是 | 显示设置，可安全恢复 |
| `window_mode` | 是 | 运行形态设置，可恢复到 `top` |
| `window_x` / `window_y` | 是 | 窗口位置可恢复 |
| `auto_start` | 是 | 通用设置，可恢复为关闭，并同步移除自启动 |
| `minimize_to_tray` | 是 | 通用设置，可恢复为开启 |
| 薪资参数 | 否 | 属于用户核心数据，避免误删 |
| 角色选择 | 否 | v0.2 只有一个正式角色时无需恢复 |

### 4.7 配置兼容

v0.2 Beta 需要在现有配置基础上新增或明确以下字段：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `debug_mode` | bool | `false` | 是否启用调试运行形态 |
| `window_mode` | string | `top` | 继续支持 `top` / `embed`，v0.2 主验收为 `top` |
| `scale` | float | `1.0` | 继续兼容 50%-200% 显示缩放 |
| `opacity` | float | `1.0` | 继续兼容 20%-100% 透明度 |
| `window_x` / `window_y` | int | `-1` | 继续兼容窗口位置记忆 |
| `auto_start` | bool | `false` | 是否启用 Windows 开机自启 |

旧版配置文件没有新增字段时，必须使用默认值运行，并在后续保存配置时补齐默认字段。配置文件加载失败时，不应导致主场景崩溃；应使用默认配置进入桌宠模式，并允许用户重新打开设置或向导。

### 4.8 平台接口与实现约束

v0.2 Beta 的 Windows 能力必须继续通过 PlatformInterface 封装，避免 Main、Settings、DragResizeSystem 直接散落 Windows 专属逻辑。

建议新增或明确以下平台接口：

| 接口 | 说明 |
|------|------|
| `setup_window(window, debug_mode)` | 根据运行模式设置窗口透明、无边框、尺寸、置顶等属性 |
| `set_mouse_passthrough(enabled, regions)` | 控制透明空白区域穿透；若平台不支持则返回失败 |
| `setup_tray(icon_path)` | 初始化系统托盘图标 |
| `update_tray_menu(visible)` | 更新托盘菜单显示/隐藏文案 |
| `set_auto_start(enabled, exe_path)` | 写入或移除开机自启配置 |
| `is_auto_start_enabled(exe_path)` | 查询当前自启动状态 |
| `is_tray_supported()` | 返回当前平台是否支持正式托盘 |

若 Godot 原生 API 无法完成托盘或穿透能力，允许在 v0.2 Beta 中引入 Windows 专用插件、GDExtension 或轻量原生桥接，但必须满足：

- 插件或扩展能随项目文档说明清楚安装/构建方式。
- 不破坏 Godot 编辑器运行主场景。
- 功能不可用时有降级路径，不影响薪资计算、Pet 和 Panel 基础显示。

### 4.9 素材 Spike

v0.2 Beta 可进行小猫动画素材探索，但不阻塞版本交付：

- 输出至少需要能被 Godot SpriteFrames 或等价资源流程接入。
- 生成质量达标时，可作为 v0.2 的附加成果接入；不达标时记录结论，继续使用当前占位素材。
- 素材 Spike 不影响桌宠窗口、输入穿透、debug 模式、托盘、自启动和设置体验的验收。

#### 4.9.1 素材验收方式

素材 Spike 的输出至少包含：

- 生成工具、输入图、提示词或工作流记录。
- idle / working / resting 三类动作的预览图或动图。
- 是否适合接入 Godot 的结论。
- 若接入，需要记录 SpriteFrames 资源路径和替换范围。

素材 Spike 成功时才进入功能实现；失败时只更新探索记录，不阻塞 v0.2 Beta 发布。

### 4.11.1 v0.2 稳定候选安全暂缓项

以下能力原本在 v0.2 范围内，但在当前 Godot 4.7 Windows 环境中反复触发原生访问违例，已从“v0.2 必须验收”调整为“v0.3 技术预研/替换实现”：

| 能力 | 当前配置 | 暂缓原因 | v0.2 替代验收 |
|------|----------|----------|----------------|
| 真系统托盘图标 | `system_tray_enabled=false` | Godot `StatusIndicator` 在导出 exe 或编辑器环境中存在访问违例风险 | 保留任务栏入口、右键菜单和退出路径；托盘接口不删除 |
| 关闭隐藏到托盘 | `minimize_to_tray=true` 但真实托盘关闭时不强制隐藏 | 托盘不可用时隐藏窗口会造成不可找回风险 | 关闭/退出不崩溃，右键菜单退出保存配置 |
| 透明无边框窗口 | `transparent_pet_window_enabled=false` | Windows 原生窗口透明能力与 Godot 4.7 组合不稳定 | 使用紧凑普通桌宠窗口，小猫和 Panel 可见 |
| 空白区域点击穿透 | `mouse_passthrough_enabled=false` | 鼠标穿透相关原生调用存在访问违例风险 | 收紧窗口尺寸，优先保证交互稳定 |

这些能力的产品价值仍然成立，不从长期规划中删除；只是 v0.2 Beta 稳定候选不再以它们作为通过/失败的硬门槛。

### 4.12 验收标准

v0.2 Beta 验收以导出 Windows exe 和 Godot 编辑器运行共同验证：

- 导出 exe 能启动并默认进入桌宠模式。
- 控制台无 parser/runtime 红色错误。
- Debug UI 默认不可见。
- `debug_mode=true` 后能进入可观察调试模式；改回 `false` 后恢复桌宠模式。
- 小猫区域 hover、单击、双击、长按、拖拽、右键菜单均可用。
- 拖拽后关闭重启，窗口位置恢复。
- 当前稳定候选中，透明空白区域点击穿透保持关闭，不因穿透能力缺失判定 v0.2 失败。
- 当前稳定候选中，真实系统托盘保持关闭，不因托盘图标缺失判定 v0.2 失败；后续 v0.3 恢复该验收。
- 点击关闭按钮和右键菜单退出不应崩溃，退出前保存配置；关闭隐藏到托盘在真实托盘恢复前暂不作为硬验收。
- 开机自启开关可写入/移除 Windows 当前用户自启动配置，设置状态与系统状态一致。
- 设置中的缩放、透明度、窗口模式、重置位置/恢复默认行为可见且不崩溃。
- M3/M4/M5 既有验证脚本继续通过，并新增 v0.2 验证脚本检查配置默认值、主场景关键节点、debug 配置兼容、平台接口、自启动配置读写和用户可见文案。

### 4.13 风险与降级方案

| 风险 | 说明 | 降级方案 |
|------|------|----------|
| 透明区域点击穿透不稳定 | Godot 透明窗口在 Windows 下可能按窗口矩形拦截输入 | 收紧窗口尺寸，保留 debug 模式，必要时用 Windows 专用实现 |
| 系统托盘 API 不稳定 | Godot 原生托盘能力可能不足或插件兼容性不确定 | 保留任务栏入口和角色右键菜单，托盘失败不阻塞主功能运行 |
| 开机自启路径不确定 | 编辑器运行时没有稳定 exe 路径 | 仅导出 exe 支持正式写入；开发态显示提示 |
| 关闭隐藏导致用户找不到窗口 | 托盘不可见或初始化失败时隐藏窗口会造成困惑 | 托盘初始化失败时关闭按钮直接退出或保留任务栏入口 |
| 自启动误写入 | 写入系统配置失败或残留旧路径 | 写入前查询，关闭时移除；验证脚本覆盖读写和清理 |
| 素材生成质量不可控 | AI 动画可能帧间不一致或不适合当前风格 | 作为非阻塞 Spike，失败只记录结论 |

## 5. v0.3 Beta PRD - 桌宠原生能力修复版

### 5.1 版本定位

v0.3 Beta 定位为**桌宠原生能力修复版**。v0.2 Beta 已经让 LetsMakeMoney 具备日常可运行的紧凑桌宠窗口、橘猫素材、薪资 Panel、设置、自启动和导出链路，但真实桌宠体验中最关键的系统能力仍因 Godot 4.7 Windows 原生访问违例而被安全关闭。v0.3 的核心目标不是继续堆叠业务功能，而是把 v0.2 暂缓的原生桌宠能力恢复到可交付状态。

v0.3 的验收重点是“整体桌宠体验成立”：应用应能以透明、无边框、可点击穿透的桌宠形态运行；小猫和 Panel 保持可交互；系统托盘成为可靠找回入口；关闭窗口时可隐藏到托盘；在托盘健康检查通过后，用户可以启用隐藏任务栏 / Alt+Tab 的纯桌宠模式。单独完成其中某一个能力不算 v0.3 达标，必须形成一条可日常使用、可恢复、可导出的闭环。

v0.3 采用 Windows x86_64 优先策略：先完成 Windows 原生能力实现，同时继续保留平台抽象接口，为后续 macOS / Linux 预留扩展边界。v0.3 不切换主引擎，不把外部辅助进程作为首选路线，优先通过 GDExtension / 原生插件修复 Godot 原生能力不足的问题。

**当前实现状态（2026-07-03）**：v0.3 Beta 已完成 Windows native bridge、真系统托盘、透明无边框窗口、点击穿透区域、关闭隐藏到托盘、纯桌宠模式门禁、设置控件修复、导出验证脚本和手动验证文档。右键菜单偶发不弹出的问题已通过“普通点击小命中框 + 右键专用上下文区域”修复。当前版本允许进入 `v0.3beta` tag 收尾；动画细节、窗口体验和设置体验的进一步打磨进入 v0.4。

| V03-G2 | 真系统托盘 | 托盘图标、托盘菜单、显示/隐藏、设置、关于、退出均可用 |

| V03-G5 | 关闭隐藏到托盘 | 关闭按钮隐藏到托盘，托盘菜单或右键菜单“退出”才真正结束进程 |

### 5.4 技术路线

v0.3 的首选技术路线是 **GDExtension / Windows 原生插件**：

- 插件负责提供 Godot 原生 API 不稳定或不足的能力，包括系统托盘、窗口样式、鼠标穿透区域、任务栏 / Alt+Tab 显示控制。
- Godot 侧继续通过 `PlatformInterface` 或等价抽象调用平台能力，不允许 Main、Settings、PetManager 等业务代码直接散落 Win32 细节。
- 插件必须随导出 exe 一起发布，用户不需要额外安装运行库或手动复制插件文件。
- Windows x86_64 是 v0.3 的唯一必须实现平台；macOS / Linux 只保留接口和降级行为，不作为验收目标。
- 外部 helper 进程不作为 v0.3 主路线；只有在 GDExtension 被验证不可行时，才作为后续技术预案记录。
- 不以切换 Godot 版本作为 v0.3 的主策略；若需要临时验证 Godot 版本差异，只能作为诊断手段，不改变项目基线。

#### 5.4.1 平台边界建议

| 层级 | 职责 |
|------|------|
| `PlatformInterface` | Godot 侧统一入口，定义托盘、窗口、穿透、自启动、纯桌宠模式等抽象能力 |
| `WindowsPlatform` | 负责 Windows 平台策略、配置读取、失败降级、插件健康检查 |
| Native Bridge / GDExtension | 负责实际 Win32 调用，不承载产品业务逻辑 |
| Main / Settings / ContextMenu | 只调用平台接口，不关心具体 Win32 API |

### 5.5 原生能力需求

#### 5.5.1 真系统托盘

系统托盘是 v0.3 的恢复入口，也是纯桌宠模式的安全前提。没有可靠托盘时，不允许默认隐藏任务栏 / Alt+Tab，也不允许关闭窗口后只隐藏到不可见状态。

托盘功能要求：

- 应用启动后创建托盘图标，图标使用当前 app icon 或专门生成的小尺寸变体。
- 托盘图标显示清晰，悬停 tooltip 至少包含 `LetsMakeMoney`。
- 左键点击托盘图标切换显示 / 隐藏桌宠窗口。
- 右键点击托盘图标打开托盘菜单。
- 托盘菜单至少包含：显示/隐藏、设置、关于 LetsMakeMoney、退出。
- 点击“设置”时，如果主窗口隐藏，应能显示主窗口或直接弹出设置窗口。
- 点击“退出”时，保存配置并真正结束进程。
- 托盘初始化失败时，不应崩溃；必须回退到保留任务栏入口和角色右键菜单的安全模式。

托盘菜单结构：

```
├── 显示/隐藏
├── 设置
├── 关于 LetsMakeMoney
├── ──────────
└── 退出
```

#### 5.5.2 透明无边框窗口

v0.3 需要恢复真实桌宠窗口形态：

- `debug_mode=false` 时，窗口应无边框、背景透明、尺寸贴合小猫和 Panel。
- 小猫、Panel、菜单、设置窗口可见且不被透明背景影响。
- 普通用户不应看到调试输入区、调试状态文字或明显的大块普通窗口背景。
- `debug_mode=true` 时仍保留 v0.2 的普通调试窗口，便于排查原生能力问题。
- 如果透明无边框初始化失败，应自动回退到 v0.2 紧凑普通窗口，并记录可读日志。

#### 5.5.3 空白区域点击穿透

点击穿透的目标是让桌宠真正像“贴在桌面上”，而不是用一个透明矩形挡住桌面。

- 小猫区域支持 hover、单击、双击、长按、拖拽、右键菜单。
- Panel 区域支持 hover 展开/收起和设置中已有交互。
- 菜单和设置窗口打开时必须可点击，不受穿透影响。
- 小猫、Panel、菜单以外的透明空白区域应尽量穿透到桌面或下层窗口。
- 缩放、透明度变化、Panel 展开/收起、窗口拖拽后，穿透区域需要重新计算或保持正确。
- `debug_mode=true` 时默认关闭点击穿透，优先保证调试可观察性。

#### 5.5.4 关闭隐藏到托盘

关闭隐藏到托盘只有在托盘健康检查通过时才启用：

- 点击窗口关闭按钮时，默认隐藏主窗口，不结束进程。
- 托盘菜单“显示/隐藏”可以恢复主窗口。
- 右键菜单或托盘菜单“退出”才真正结束进程。
- 退出前保存窗口位置、设置状态和薪资配置。
- 如果托盘初始化失败，关闭按钮不得把应用隐藏到无法找回的位置，应直接退出或保持任务栏入口。

#### 5.5.5 可找回优先的纯桌宠模式

v0.3 引入纯桌宠模式，但必须以“可找回”为第一原则。

- 默认模式仍保留任务栏和 Alt+Tab 入口。
- 当且仅当托盘图标创建成功、托盘菜单可用、显示/隐藏能力通过健康检查后，设置中才允许启用纯桌宠模式。
- 纯桌宠模式开启后，可以隐藏任务栏入口和 Alt+Tab 入口，让应用更像真正桌宠。
- 纯桌宠模式中，托盘是主要找回入口，角色右键菜单是辅助入口。
- 如果托盘在运行中异常、插件健康检查失败或配置异常，应自动退出纯桌宠模式，恢复任务栏 / Alt+Tab 入口。
- 设置页必须清楚表达当前是否处于纯桌宠模式，以及为什么某些情况下不能开启。

### 5.6 设置体验与配置

v0.3 不重做完整设置系统，但需要修正 v0.2 中表达不清的模式类控件，并为原生能力提供可理解的状态反馈。

#### 5.6.1 设置体验要求

| 设置项 | v0.2 状态 | v0.3 要求 |
|--------|-----------|-----------|
| 休息模式 | 交互表达不够清晰 | 使用下拉框、分段控件或等价明确控件，不再像普通开关一样含糊 |
| 窗口模式 | CheckButton 表达 `top/embed` 不直观 | 使用下拉框或分段控件展示“置顶悬浮 / 融入桌面 / 纯桌宠模式”之间的关系 |
| 透明度 | 已显示百分比 | 保持百分比显示，确保保存后立即反馈 |
| 缩放 | 已显示百分比 | 保持百分比显示，缩放后同步更新窗口和穿透区域 |
| 纯桌宠模式 | v0.2 无 | 新增设置入口，但仅在托盘健康检查通过时允许开启 |
| 原生能力状态 | v0.2 仅靠日志和配置 | 增加只读状态提示，例如托盘：可用/不可用，点击穿透：可用/已降级 |
| 重置窗口位置 | 已完成 | 保留，并作为纯桌宠模式下的重要恢复手段 |

#### 5.6.2 配置字段建议

v0.3 需要在兼容 v0.2 配置的基础上新增或明确以下字段：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `native_integration_enabled` | bool | `true` | 是否允许启用原生插件能力；故障排查时可关闭 |
| `system_tray_enabled` | bool | `true` | 是否启用真系统托盘；插件不可用时自动降级 |
| `transparent_pet_window_enabled` | bool | `true` | 是否启用透明无边框桌宠窗口 |
| `mouse_passthrough_enabled` | bool | `true` | 是否启用透明空白区域点击穿透 |
| `pure_pet_mode` | bool | `false` | 是否隐藏任务栏 / Alt+Tab；必须依赖托盘健康检查 |
| `minimize_to_tray` | bool | `true` | 点击关闭按钮时是否隐藏到托盘 |
| `debug_mode` | bool | `false` | 继续保留 v0.2 debug 入口 |

旧配置缺少这些字段时，必须按默认值处理。若用户从 v0.2 升级到 v0.3，配置迁移不应破坏薪资、窗口位置、缩放、透明度、自启动等既有设置。

### 5.9 验收标准

v0.3 Beta 需要以 Godot 编辑器运行和导出 Windows exe 双路径验证，最终验收以导出 exe 为准。

- 导出 exe 能正常启动，不再出现与托盘、透明窗口、穿透相关的 native read 访问违例。
- 原生插件随 exe 一起发布，用户不需要额外安装或手动复制插件。
- `debug_mode=false` 时默认进入桌宠运行形态，小猫和 Panel 可见。
- 透明无边框窗口可用，普通用户不看到明显普通窗口背景。
- 透明空白区域点击尽量穿透到桌面或下层窗口。
- 小猫区域 hover、单击、双击、长按、拖拽、右键菜单继续可用。
- Panel 折叠 / 展开、边缘定位、薪资显示继续可用。
- 托盘图标可见，左键显示/隐藏、右键菜单、设置、关于、退出均可用。
- 点击关闭按钮时，在托盘健康状态下隐藏到托盘；可通过托盘恢复。
- 托盘不可用时，应用自动回退到可找回模式，不隐藏到不可恢复状态。
- 默认保留任务栏 / Alt+Tab；开启纯桌宠模式后隐藏任务栏 / Alt+Tab，并能通过托盘找回。
- 原生插件健康检查失败时，纯桌宠模式自动关闭或禁止开启。
- 休息模式、窗口模式等设置控件表达清晰，不再出现“像开关但实际是模式选择”的歧义。
- v0.3 手动验证文档、progress、release notes / changelog 均更新。

### 5.10 风险与降级方案

| 风险 | 说明 | 降级方案 |
|------|------|----------|
| GDExtension 引入新崩溃 | 原生插件调用 Win32 API 时可能再次触发访问违例 | 所有原生能力必须有健康检查；失败时关闭插件能力，回退 v0.2 紧凑窗口 |
| 插件打包遗漏 | 导出 exe 缺少动态库或资源文件会导致用户机器不可用 | 导出验证脚本检查插件文件存在，并在 release notes 中列出分发文件 |
| 托盘不可用导致应用丢失 | 关闭隐藏或纯桌宠模式依赖托盘找回 | 托盘健康检查未通过时禁止隐藏任务栏 / Alt+Tab，关闭按钮不隐藏到托盘 |
| 点击穿透区域错误 | 误把小猫或 Panel 区域穿透，导致交互失效 | Debug 模式默认关闭穿透；提供日志和可视化验证步骤；必要时回退为收紧窗口 |
| 透明窗口影响设置窗口 | 穿透或透明可能误影响设置对话框 | 设置窗口始终使用普通可点击窗口，不参与穿透区域 |
| 文档整理变成删减 | 历史文档很详细，过度压缩会损失上下文 | 只清理编码、结构和重复表达，不删除关键设计依据、验收记录和风险说明 |

### 5.11 v0.4 前置说明

橘猫动画打磨不进入 v0.3。v0.3 只保证现有橘猫素材在真实桌宠窗口中显示完整、缩放正确、交互状态不回退。动画表现、状态过渡、更多动作反馈、窗口细节和整体使用体验统一进入 v0.4 Beta 大型优化版本。


## 6. v0.4 Beta PRD - 大型体验优化版（草案）

### 6.1 版本定位

v0.4 Beta 定位为**大型体验优化版**。v0.3 已经把 LetsMakeMoney 从“紧凑普通窗口”推进到更接近真实桌宠的状态：有原生托盘、透明无边框窗口、点击穿透、关闭隐藏到托盘、纯桌宠模式门禁和导出验证链路。v0.4 不再以补齐底层能力为主，而是把这些能力打磨成更自然、更稳定、更适合日常常驻桌面的体验。

v0.4 的核心目标是让用户不再明显感到“这是一个被调出来的 Godot 窗口”，而是感觉桌面上确实有一只会赚钱、会反馈、能被轻松找回和配置的小猫。版本重点包括橘猫动画、状态切换、点击/长按/拖拽反馈、窗口穿透区域、边缘定位、设置体验、保存反馈、性能与发布体验。

v0.4 的默认体验方向采用**温暖陪伴型**，并进一步落为**极简生产力小工具 + 宠物陪伴**：应用需要保留“挣钱工具”的核心价值，但整体观感不应变成压迫式效率仪表盘。橘猫、Panel、设置页和反馈文案应更偏柔和、可靠、轻量陪伴，让用户愿意长期把它放在桌面上。v0.4 可以统一默认视觉语言、动效节奏和反馈层次，但不实现“多主题/主题切换/主题商店”等主题系统能力；主题系统作为后续独立模块规划。

### 6.2 当前已知问题汇总

以下问题来自 v0.3 开发和手动验证收尾阶段，不阻塞 `v0.3beta`，但应作为 v0.4 的主要输入：

| 分类 | 当前表现 | v0.4 处理方向 |
|------|----------|----------------|
| 动画质量 | 橘猫素材已接入，但 idle / working / resting / clicked_hold 以及基础状态延伸的单击/双击动作仍偏基础，帧数、节奏和状态差异不够细腻 | 建立动画规格、素材命名、帧边界、锚点和循环节奏标准，打磨基础状态与交互延伸动画 |
| 交互反馈 | 单击、双击、长按、拖拽、右键菜单均已可用，但反馈层次仍偏功能性，长按和拖拽之间的体验边界需要继续调优 | 细化交互优先级、反馈时长、恢复状态规则和可感知动画 |
| 点击穿透 | v0.3 使用交互矩形 + native hit-test，已解决核心点击穿透和右键菜单问题，但仍不是像素级命中，边界需要更多真实场景校准 | 增加 debug 可视化、命中区测试点和多缩放/DPI 验证，逐步收敛到更自然的命中范围 |
| Panel 体验 | Panel 折叠/展开可用，薪资显示正常，但与小猫距离、边缘吸附、展开方向和窗口边界关系仍可优化 | 优化 Panel 定位策略、边缘安全区、展开动画和内容密度 |
| 设置体验 | v0.3 已修复下拉框、保存、原生能力状态，但设置窗口仍偏工程化，反馈、分组、恢复入口和错误提示不够顺滑 | 重构设置信息架构，强化即时反馈、保存状态、恢复默认和错误提示 |
| UI polish | v0.4 已新增 UI polish spec 和可交互原型，但 PRD 中尚未把它定义为正式需求模块 | 将 UI polish 与交互原型纳入 v0.4 正式范围，明确 Panel 折叠态“金额 + 短状态”、设置窗口工具化、交互说明和验收门禁 |
| 纯桌宠模式 | 已有可找回门禁，但用户对任务栏/Alt+Tab/托盘之间关系仍可能困惑 | 增加模式说明、状态提示和异常恢复路径，避免用户误以为窗口丢失 |
| 构建发布 | native DLL、exe、导出脚本已可用，但当前仍是本地脚本式发布，没有安装器、发布包清单和用户侧升级说明 | 规范 release 包结构、分发清单、版本号、升级/回滚说明 |

### 6.4 动画与素材需求

v0.4 需要把橘猫动画作为正式优化模块，而不是继续停留在“素材已接入”的状态。

- 基础状态仍保持 `idle`、`working`、`resting` 三类，不再把单击、双击、长按作为独立基础状态。
- 单击、双击是当前基础状态的延伸动作，例如 `idle_clicked_single`、`working_clicked_double`、`resting_clicked_single`；长按继续使用通用 `clicked_hold` 持续反馈。
- 单击、双击、长按播放结束后都应回到进入交互前的基础状态。
- 每套动画需要统一画布尺寸、角色锚点、视觉中心、脚底基线、缩放倍率和透明边界。
- `working` 需要具有明确“正在赚钱/敲键盘/努力工作”的动作识别度，并允许加入键盘、电脑、金币等工作/金钱道具。
- `resting` 需要与 idle 明显区分，避免用户看不出当前处于非工作时间；v0.4 素材筛选阶段需要同时探索困倦坐姿、躺下和其他低动作休息姿态。
- `clicked_hold` 需要支持长按持续状态，松开后自然恢复，不影响拖拽判定。
- 动画帧接入前应先在预览文档或验证场景中确认帧间不漂移、不裁切、不闪烁。
- 若继续使用 SpriteCook 或同类生成流程，应记录输入图、提示词、输出素材、筛选结论和接入成本。

#### 6.4.1 素材生成产能路线

v0.4 不应把橘猫 v2 的素材产出完全押在单一免费额度或单一外部服务上。动画模块需要同时记录短期可用、长期可控和确定性兜底三类方案：

| 路线 | 定位 | v0.4 要求 |
|------|------|-----------|
| SpriteCook | 短期快速候选生成工具 | 可用于生成第一批 v2 候选和动画探索，但必须记录额度、输入、提示词、输出和筛选结论，不把免费额度视为长期唯一产能 |
| ComfyUI / 本地扩散工作流 | 中长期自托管产能 Spike | 作为 v0.4 非阻塞 Spike 记录：评估参考图保持、姿态控制、透明背景、关键帧或短动画生成可行性，暂不作为 v0.4 发布阻塞 |
| Pixelorama / LibreSprite / Aseprite 等编辑器 | 人工修边与低帧整理工具 | 用于清理透明边界、统一画布、修正漂移和整理逐帧素材；优先选择免费开源工具，Aseprite 作为体验较好的可选工具 |
| Godot cutout / AnimationPlayer | 长期确定性兜底方案 | 当 AI 逐帧动画不稳定时，允许将小猫拆为头、身体、尾巴、爪子、眼睛、键盘、电脑、金币等部件，用 Godot 动画实现呼吸、眨眼、打字、金币跳动等稳定动作 |

橘猫 v2 是否替换默认素材，仍以动画质量门禁为准。素材产能路线的调研结论需要记录在 `doc/v0.4-animation-assets-log.md`，并在接入前确认当前选择的是“生成逐帧”“人工修帧”还是“cutout 动画”。

### 6.5 使用体验需求

v0.4 需要把 v0.3 中“能用”的交互整理成“好用”的交互。

- Hover 状态应轻量，不打断基础动画太久。
- 单击反馈应短促明确，不影响快速继续拖拽或右键。
- 双击反馈应能被稳定识别，不被两次单击吞掉。
- 长按反馈应与拖拽阈值协调，避免用户想拖动时先进入长按状态。
- 拖拽移动继续以鼠标屏幕位移为准，不能出现窗口移动速度异常。
- 右键菜单在小猫头部、身体、Panel 附近的合理区域都应稳定弹出。
- 关闭隐藏、托盘显示/隐藏、纯桌宠模式之间需要更明确的用户反馈。
- 设置保存后应有明确状态提示，避免用户不知道保存是否完成。

### 6.6 窗口与 Panel 需求

v0.4 继续以 Windows x86_64 为主要验证平台，重点优化真实桌宠窗口的日常观感。

- 点击穿透需要提供可调试视图或日志模式，便于确认当前交互矩形。
- 小猫命中区、右键上下文区、Panel 命中区需要区分普通点击、右键、hover、拖拽等场景。
- 多缩放比例下，小猫和 Panel 不能裁切、错位或出现不可点击区域。
- Panel 在屏幕右侧、底部、左侧等边缘位置应自动选择更合理的展开方向。
- 纯桌宠模式下仍要优先保证可找回：托盘异常、native 失败、配置异常时自动恢复任务栏入口。
- Debug 模式应能帮助定位窗口尺寸、命中区和 Panel 位置，但默认用户模式不显示调试 UI。

### 6.7 设置、验证与发布需求

- 设置页需要继续保留 Salary、Pet、Display、Panel、General 的大类，但可以优化具体布局和说明文案。
- Display 页需要更清楚解释窗口模式、纯桌宠模式、透明度、点击穿透之间的关系。
- General 页需要保留开机自启、关闭隐藏到托盘、Debug 模式和恢复入口。
- 配置保存应尽量即时、轻量，并避免重复写注册表或重复调用 native 接口。
- v0.4 关键体验模块在实现前需要有原型或设计稿支撑，至少覆盖默认桌宠形态、Panel 展开/收起、设置页关键状态和发布包说明页。
- 手动验证文档需要覆盖动画、交互、窗口、Panel、设置、托盘、发布包和升级路径。
- 自动验证继续覆盖 v0.1/v0.2/v0.3 回归项，新增 v0.4 动画资源、命中区模型、设置保存反馈和发布包清单检查。
- Release Notes 需要明确 v0.4 对用户可感知的体验提升，以及仍未进入范围的能力。

### 6.8 UI polish 与交互原型需求

v0.4 将 UI polish 作为正式体验需求，而不是独立主题系统。该模块的目标是把“极简生产力小工具 + 宠物陪伴”落实到默认 Panel、设置窗口、反馈文案和可交互原型中，让后续 Godot UI 实现有明确对照。

#### 6.8.1 UI polish 定位

- UI polish 不改变 LetsMakeMoney 的核心产品定位：小猫负责陪伴感，薪资计算和 Panel 负责生产力反馈。
- Panel / 桌宠主界面可以更温暖、更轻量，但必须保持薪资信息可读。
- 设置窗口保持工具化、清楚、可恢复，不做宠物主题化包装。
- v0.4 不新增 Theme Tab，不新增外观切换入口，不提供多主题能力。
- UI polish 的主规格文档为 `doc/v0.4-ui-polish-spec.md`，原型入口为 `doc/prototypes/index.html`。

#### 6.8.2 Panel polish 要求

- 折叠态 Panel 必须从“单独金额”升级为“金额 + 短状态”，例如 `¥12.34 工作中`。
- 折叠态不显示进度条、今日/本月/时薪等明细，避免常驻区域变成密集仪表盘。
- 展开态保留完整薪资信息，但信息层级应调整为：今日已赚优先，当前状态辅助判断，本月累计/时薪/工作进度作为详情。
- Panel 的默认视觉应像轻量薪资便签，避免压迫式效率仪表盘观感。
- Panel 动效应保持轻量、可中断，不通过强烈缩放或夸张弹跳抢小猫主体。

#### 6.8.3 设置窗口 polish 要求

- 设置窗口继续保留 Salary / Pet / Display / Panel / General 五个设置分类，但 v0.4 的视觉参考应从旧顶部小 Tab 过渡到 Windows 11 设置窗口式布局：左侧分类导航、右侧内容区、卡片式设置项、清晰标题层级。
- Windows 11 设置窗口是风格参考，不要求逐像素复刻系统设置，也不引入系统账号、系统设备信息等无关内容。
- Salary / Pet / Display / Panel / General 的职责需要在原型和文档中保持清楚，不把外观主题配置混入 v0.4。
- 透明度、缩放、窗口模式、纯桌宠、点击穿透、自启动、Debug、关闭隐藏到托盘、恢复默认等配置应在对应页签中有可理解说明。
- 保存反馈需要覆盖：已保存、无变化、需要重新显示窗口/重启生效、保存失败。
- 保存反馈应显示在设置窗口内部，不依赖可能被桌宠窗口遮挡的外部弹层。
- 恢复默认设置需要明确影响范围，不默认清空薪资、角色和核心用户数据。
- 设置窗口需要保留一个清晰的搜索入口或预留搜索区域；若 v0.4 不实现真实搜索，原型中应标注为占位，不影响主要设置操作。

#### 6.8.4 右键菜单与快捷入口要求

- 右键小猫菜单不再把“设置窗口模式”和“选择宠物”作为平铺快速入口。
- 菜单应采用二级入口：鼠标悬停一级项时，在右侧展开对应可选项。
- `窗口模式` 二级菜单至少包含 `置顶悬浮`、`融入桌面`；当前选中项需要有勾选或高亮反馈。
- `选择宠物` 二级菜单至少列出当前可用宠物；只有一个宠物时仍展示当前项，但应禁用不可用切换或显示当前标记，不报错。
- 一级菜单保留 `设置`、`关于 LetsMakeMoney`、`退出` 等明确命令。
- 二级菜单展开不得破坏右键菜单可点击性、点击穿透、右键上下文区和托盘降级入口。

#### 6.8.5 图标 polish 要求

- v0.4 需要统一应用窗口图标、托盘图标、导出 exe 图标和关于窗口中的应用图标。
- 图标应基于当前橘猫形象优化，保证 16 / 24 / 32 / 48 / 64 / 128 / 256 尺寸下都清晰，不依赖简单缩放导致糊边。
- 托盘图标需要在 Windows 深色/浅色任务栏中保持可识别，必要时提供更简化的小尺寸轮廓版本。
- 窗口图标与托盘图标可以同源，但允许针对托盘小尺寸单独优化。
- 发布包需要包含最终图标资源，并确保 Godot export preset 使用同一套图标。

#### 6.8.6 可交互原型要求

v0.4 原型必须不仅是静态截图，还要支持用于评审的轻交互：

- 左侧导航可切换 v0.1-v0.4 各原型屏。
- v0.4 UI polish 屏可切换 Panel 的 `对照 / 折叠 / 展开` 状态。
- v0.4 UI polish 屏可切换 `工作中 / 休息中 / 待配置` 状态，用于观察金额、短状态和进度变化。
- v0.4 UI polish 屏可切换设置反馈的 `已保存 / 无变化 / 需重显 / 保存失败` 状态。
- v0.4 UI polish 屏的设置预览可点击 Salary / Pet / Display / Panel / General 分类，观察不同设置页的信息组织。
- v0.4 UI polish 屏需要展示 Windows 11 设置窗口式的左侧导航、右侧设置卡片、标题层级和搜索入口占位。
- v0.4 UI polish 屏需要展示右键菜单二级入口：`窗口模式` 与 `选择宠物` 悬停展开右侧子菜单。
- v0.4 UI polish 屏需要展示应用图标/托盘图标的目标观感或尺寸说明。
- 原型页面必须包含交互说明，明确每组按钮用于验证什么，不让评审者猜交互意图。
- 原型图片资源必须使用项目内真实存在的素材路径，不能引用已失效的历史目录。

#### 6.8.7 UI polish 验收标准

- `doc/v0.4-ui-polish-spec.md` 存在，并明确“极简生产力小工具 + 宠物陪伴”的定位。
- `doc/prototypes/index.html` 存在可点击的 v0.4 UI polish 原型屏。
- 原型中的 Panel 折叠态显示“金额 + 短状态”，展开态显示完整薪资详情。
- 原型中的工作状态切换能改变金额、短状态和进度表达。
- 原型中的设置反馈切换能展示已保存、无变化、需重显和保存失败四类反馈。
- 原型中的设置预览参考 Windows 11 设置窗口结构，保持工具化，不加入 Theme Tab、不加入主题切换、不加入宠物装饰化设置。
- 原型中的右键菜单展示 `窗口模式`、`选择宠物` 的二级展开形式，不再平铺快速入口。
- 原型或图标规格中明确托盘图标、窗口图标、导出 exe 图标的尺寸和清晰度要求。
- 原型说明文档 `doc/prototypes/prototype-spec.md` 同步说明交互方式和验收标准。
- 后续 Godot UI 实现必须对照该原型，不应在实现阶段重新发明一套与 PRD 相冲突的视觉/交互方向。

### 6.9 范围外

v0.4 Beta 暂不做以下内容：

- macOS / Linux 原生托盘、透明窗口、点击穿透实现。
- iOS / Android 移动端。
- 多角色同屏、角色商店、账号系统、云同步。
- 多主题、主题切换、主题商店、用户自定义主题。
- 完整统计报表、收入趋势图、日历热力图。
- 彻底重写 Godot 架构或替换主引擎。

### 6.10 验收标准

- v0.4 导出 exe 能启动并稳定常驻，无 native read/write 访问违例。
- 橘猫 idle / working / resting / clicked_hold 以及 idle/working/resting 下单击、双击延伸动画可见、不卡死、不裁切、不明显漂移。
- 单击、双击、长按、拖拽、右键菜单在真实桌宠窗口中均可稳定复现。
- 小猫和 Panel 外透明区域在常见场景下继续点击穿透。
- Panel 边缘定位、展开/收起和薪资显示在多缩放比例下保持可读。
- Panel 折叠态符合 UI polish 要求，显示“金额 + 短状态”，展开态符合轻量薪资便签层级。
- 设置项保存反馈清晰，进入任意选项卡后不影响其他选项卡保存。
- 设置窗口整体风格参考 Windows 11 设置窗口：左侧分类导航、右侧卡片式内容、清晰搜索入口或搜索占位、稳定的保存反馈。
- 右键小猫菜单中的 `窗口模式` 和 `选择宠物` 使用二级菜单展开，不再作为平铺快速入口。
- 托盘图标、窗口图标和导出 exe 图标清晰统一，在常见 Windows 深色/浅色环境中可识别。
- UI polish 原型已支持 Panel 状态、业务状态、设置反馈和设置 Tab 交互，并在文档中说明交互目的。
- 设置窗口 polish 保持工具化，不加入主题系统入口或宠物装饰化设置。
- 托盘显示/隐藏、关闭隐藏到托盘、纯桌宠模式仍保持可找回。
- v0.4 PRD、Implementation Plan、Progress、Manual Verification、Release Notes 均与实际开发状态同步。

### 6.11 后续规划：主题系统

主题系统是有效的长期方向，但它的产品性质与 v0.4 的“大型体验优化”不同。v0.4 需要先把默认体验打磨到稳定、舒服、可长期使用，再考虑把视觉风格开放为可切换模块。

后续主题系统可以独立规划为以下能力：

- 主题资源模型：颜色、Panel 样式、设置页样式、动效节奏、图标和可能的角色皮肤需要有统一配置结构。
- 默认主题继承：所有新增主题必须继承 v0.4 打磨后的默认交互和可用性，不允许因为主题变化破坏点击穿透、Panel 定位或设置保存。
- 主题切换入口：主题选择适合放在设置页的外观/显示相关分组中，但需要避免和窗口模式、透明度、缩放等功能设置混在一起。
- 主题验证标准：每套主题都需要验证浅色/深色桌面背景、Panel 可读性、透明窗口边界、托盘/设置入口一致性。
- 主题交付节奏：主题系统不作为 v0.4 Beta 的通过条件，可在 v0.5 或后续独立版本中作为完整模块设计。
