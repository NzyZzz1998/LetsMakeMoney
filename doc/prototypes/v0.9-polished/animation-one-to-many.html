Runtime contract

运行时不再按固定 1.55 秒强制返回

01解析基础状态working / awake_rest / sleeping
→
02选择语义动作输入优先级、冷却、环境与事件
→
03按 manifest 播放逐帧时长、锚点、命中轮廓
→
04完成或超时忽略旧 token 的晚到事件
→
05重新解析恢复最新状态，不恢复过期缓存

中断优先级

Modal / 窗口关闭 ＞ 长按拖拽跑动 ＞ 状态感知单击 ＞ 业务事件 ＞ 环境动作 ＞ 基础循环。双击不再拥有独立产品动作。

动作缺失回退

状态专属动作 → 通用同语义动作 → 当前基础循环。不得按 pet_id 写一次性特判。

超时保护

总时长 + max(500ms, 25%)。超时只恢复状态并记录异常，不截断正常长动作。

Decision gates

建议审核结论与实施顺序

  1. P0
    先完成 Classic 的分层动作合同审计

    已有动作明确复用；真正缺失的工作、单击和方向跑动动作进入生成队列。

  2. P0
    先接运行时合同，再替换默认宠物

    逐帧时长、锚点、动作完成、超时和回退全部通过后，Classic 才能成为默认。

  3. P1
    Classic 验收通过后再补多多

    复用动作名称、触发与测试，不复用图片，也不降低多多身份一致性。

  4. P1
    最后联调动态命中与点击穿透

    大幅动作必须更新透明轮廓，Panel、菜单和拖拽仲裁不得回退。

推荐方案

状态合同先行，Classic 精做，多多复用合同后补图

这样可以先解决“动作不流畅且看不出区别”的核心问题，同时避免一次生成两套仍然需要返工的素材。
