# LetsMakeMoney Prototype Spec

**版本**: v0.1 Beta / v0.2 Beta / v0.3 Beta / v0.4 Beta 原型说明
**日期**: 2026-07-04
**原型入口**: `doc/prototypes/index.html`

---

## 1. 原型目标

本原型用于补齐 LetsMakeMoney 在 v0.1 到 v0.4 期间缺失的界面设计表达。它不是 Godot 场景实现文件，而是一个用于产品 review、工程对照和后续 UI 实现讨论的交互原型。v0.1-v0.3 部分保持中保真工程对照风格；v0.4 部分开始承接大型体验优化版本的高保真方向确认。v0.4 UI polish 的主规格见 `doc/v0.4-ui-polish-spec.md`。

原型重点回答这些问题：

- v0.1 调试窗口版、v0.2 紧凑桌宠候选、v0.3 真桌宠修复版、v0.4 大型体验优化版的界面关系是什么。
- 桌宠主窗口、薪资 Panel、右键菜单、设置窗口、首次启动向导、系统托盘、Debug 模式、纯桌宠模式分别长什么样。
- v0.3 的真托盘、透明无边框、点击穿透、关闭隐藏到托盘和纯桌宠模式如何被用户理解。
- v0.4 的默认温暖陪伴型体验、Panel 边缘状态、设置反馈、托盘找回和正式 Zip beta 发布包应如何表达。
- v0.4 的“极简生产力小工具 + 宠物陪伴”如何落到 Panel、设置窗口、文案和动效规格。
- 每个界面对应哪些配置字段、验收点和降级条件。

## 4. 视觉方向

v0.1-v0.3 原型采用偏工具软件的中保真风格：

- 以浅灰、白色、深色文字和状态色为主。
- 小猫保留真实素材占位，但不把整体做成游戏/玩具风。
- 设置、Debug、状态说明和降级逻辑优先清晰。
- 所有界面带右侧开发注释栏，便于对照 PRD、实施计划和验收标准。

v0.4 默认体验方向采用**温暖陪伴型**：

- 视觉、动效、Panel 和文案应柔和、轻量、低压力。
- 仍然保留“薪资工具”属性，金额、工作状态、进度和设置入口不能被弱化成纯装饰。
- Panel 与小猫的关系要像桌面陪伴物，而不是压迫式效率仪表盘。
- 设置说明要清楚、可恢复、可理解，避免工程化黑话。
- 不提供外观切换入口，不把默认体验包装成可选主题。

v0.4 UI polish 进一步明确为**极简生产力小工具 + 宠物陪伴**：

- 小猫负责陪伴感、情绪反馈和桌面存在感。
- Panel 负责轻量薪资反馈，折叠态显示金额和短状态，例如 `¥12.34 工作中`。
- 设置窗口继续偏工具化，v0.4 UI polish 屏参考 Windows 11 设置窗口，采用左侧分类导航、右侧卡片式设置项和搜索入口占位，不做可爱化主题包装。
- 右键菜单在 v0.4 UI polish 屏中采用二级入口，`窗口模式` 与 `选择宠物` 不再平铺为快速项。
- 托盘、窗口、exe 和关于窗口图标在 v0.4 UI polish 屏中展示多尺寸预览，强调小尺寸托盘图标需要单独优化。
- 详细 token、组件、文案和验收标准以 `doc/v0.4-ui-polish-spec.md` 为准。

## 5. 交互说明

`index.html` 支持轻交互：

- 左侧导航切换原型屏。
- “切换 Panel 展开”按钮可在折叠/展开 Panel 间切换。
- “切换设置 Tab”按钮可轮换设置页。
- 设置窗口的 Tab 可点击切换。
- 右侧注释栏会随当前屏更新版本来源、配置字段、验收点和降级风险。
- v0.4 新增屏幕用于确认默认体验、Panel 边缘策略、设置反馈、UI polish 和发布包说明。
- v0.4 UI polish 屏支持切换 Panel 的“对照 / 折叠 / 展开”状态。
- v0.4 UI polish 屏支持切换“工作中 / 休息中 / 待配置”状态，用于观察金额、状态文案和进度条变化。
- v0.4 UI polish 屏支持切换设置反馈的“已保存 / 无变化 / 需重显 / 保存失败”状态。
- v0.4 UI polish 屏的工具化设置预览支持点击左侧 Salary / Pet / Display / Panel / General 分类，观察不同设置页的信息组织。
- v0.4 UI polish 屏支持切换右键菜单中的 `窗口模式 >` 与 `选择宠物 >` 二级菜单。
- v0.4 UI polish 屏展示 16 / 24 / 32 / 48 / 64 / 128 / 256 图标尺寸预览，用于确认托盘和窗口图标优化方向。

这些交互用于 review 产品结构，不代表最终 Godot 交互实现。

## 7. 关键设计决策

### 7.1 默认仍保留可找回入口

v0.3 虽然要实现纯桌宠模式，但原型明确表达：默认状态仍保留任务栏 / Alt+Tab。只有托盘健康检查通过后，用户才能手动开启纯桌宠模式。

### 7.2 透明空白区域不是交互区

桌宠主窗口和纯桌宠状态都用虚线标注透明空白区域。该区域在 v0.3 中应尽量穿透到桌面或下层窗口。只有小猫、Panel、菜单和设置窗口属于交互区域。

### 7.3 右键菜单是重要降级入口

即使 v0.3 实现真系统托盘，角色右键菜单仍保留“设置 / 关于 / 退出”等入口。托盘不可用时，右键菜单是用户找回和退出应用的重要降级路径。

### 7.4 设置窗口偏工具化

设置窗口不追求强装饰。它承担配置、诊断、原生能力状态反馈和恢复操作，因此原型中强调：

- 明确选择控件。
- 原生能力状态。
- 保存失败不显示假成功。
- 恢复默认不清薪资配置。

### 7.5 v0.4 默认体验不是主题系统

v0.4 的温暖陪伴型是默认体验方向，不是可切换主题能力。本版本只统一默认视觉、动效、Panel、设置和反馈的表达方式。

进入后续独立规划的能力：

- 多主题。
- 主题切换。
- 主题商店。
- 用户自定义主题。

v0.4 原型中如果需要表达外观相关内容，只能作为默认体验说明，不提供用户选择入口。

### 7.6 v0.4 UI polish 决策

v0.4 UI polish 采用“极简生产力小工具 + 宠物陪伴”。该决策用于约束 v0.4 后续原型和 Godot UI 实现：

- Panel / 桌宠主界面可以更温暖、更轻量。
- 设置窗口保持偏工具化，不可爱化；v0.4 UI polish 原型中参考 Windows 11 设置窗口，采用左侧分类导航、右侧卡片式设置项和搜索入口占位。
- 折叠态 Panel 显示“金额 + 短状态”，不显示进度条。
- 展开态 Panel 保留完整薪资信息，但层级从报表式调整为轻量便签式。
- 保存反馈、无变化反馈、失败反馈和需要重启/重显说明都应在设置窗口内清晰表达。
- 右键菜单使用二级入口承载窗口模式和选择宠物，一级菜单保留清晰命令。
- 图标 polish 需要同时关注托盘 16/24 小尺寸和窗口/exe 大尺寸，不直接依赖简单缩放。
- 不新增 Theme Tab，不新增外观切换入口。

详细规格见 `doc/v0.4-ui-polish-spec.md`。

### 7.7 v0.4 Panel 体验原则

- 折叠态金额需要垂直居中。
- 折叠态显示金额和短状态，例如 `¥12.34 工作中`。
- 展开态优先保证今日已赚、本月累计、时薪、进度和状态可读。
- 小猫靠右时 Panel 向左展开。
- 小猫靠左时 Panel 向右展开。
- 小猫靠底时 Panel 向上展开。
- 四角场景优先保证 Panel 完整可见，再保证距离小猫较近。
- Panel 不贴脸、不离太远、不遮挡小猫主体。

### 7.8 v0.4 设置体验原则

- 设置页继续保留 Salary / Pet / Display / Panel / General 五个分类。
- v0.4 UI polish 原型采用 Windows 11 设置窗口式布局：左侧分类导航、右侧卡片式设置项、顶部搜索入口或禁用占位。
- Display 页解释窗口模式、透明度、缩放、纯桌宠模式和点击穿透之间的关系。
- General 页解释开机自启、关闭隐藏到托盘、Debug 模式、重置窗口位置和恢复默认设置。
- 保存成功、保存失败、无变化、平台能力不可用都需要明确反馈。
- 设置窗口应呈现为单一窗口或明确的独立对话框，不再出现大宿主窗口套小设置窗的观感。
- 设置窗口保持工具化，不加入猫咪装饰、主题选择或可爱化皮肤。

### 7.9 v0.4 右键菜单与图标原则

- 右键小猫菜单一级项保留设置、窗口模式、选择宠物、关于和退出。
- 窗口模式与选择宠物使用二级菜单，鼠标悬停或原型按钮可切换右侧子菜单。
- 当前窗口模式和当前宠物必须有勾选、高亮或当前标记，不能只靠颜色表达。
- 只有一个宠物时仍展示当前项，不因没有可切换角色而报错。
- 托盘图标、窗口图标、exe 图标和关于窗口图标保持同源识别。
- 16 / 24 小尺寸图标允许使用简化猫头或轮廓版本，避免完整身体缩小后不可读。

### 7.10 v0.4 发布包体验原则

v0.4 发布形态是 Zip beta 包，不做安装器和自动更新。

Zip 包原型清单：

```text
LetsMakeMoney.exe
letsmakemoney_native.dll
README.md
v0.4-beta-notes.md
manifest.json
checksums.txt
```

发布说明需要解释配置路径、Debug 模式入口、已知限制和问题回报方式。

## 8. 后续使用方式

建议在实现 v0.3 时按以下方式使用原型：

1. 开发 V03-M1 前，用产品总览和右侧注释确认原生能力范围。
2. 开发 V03-M2 前，对照系统托盘屏确认托盘菜单和找回路径。
3. 开发 V03-M3 / V03-M4 前，对照桌宠主窗口和纯桌宠状态确认透明、穿透、交互区域。
4. 开发 V03-M5 前，对照纯桌宠状态确认“可找回优先”。
5. 开发 V03-M6 前，对照设置窗口确认 OptionButton、状态提示和恢复默认范围。
6. 手动验证前，对照原型和 `doc/verification/v0.3.md` 检查用户可见体验是否一致。

建议在实现 v0.4 时按以下方式使用原型：

1. 开发 V04-M0 时，先确认 v0.4 四个新增原型屏是否覆盖默认桌宠、Panel、设置、托盘与发布包。
2. 开发 UI polish 前，对照 `doc/v0.4-ui-polish-spec.md` 确认 Panel、设置、文案和动效边界。
3. 开发 V04-M1 前，对照 v0.4 默认桌宠体验确认橘猫动画方向，不直接把不达标素材接入默认资源。
4. 开发 V04-M2 / V04-M3 前，对照 Panel 边缘状态和默认桌宠体验确认 hover、点击穿透和命中区边界。
5. 开发 V04-M4 前，对照设置关键状态确认保存反馈、恢复默认和平台禁用说明。
6. 开发 V04-M5 前，对照托盘与发布包屏确认 Zip 包清单和找回说明。

- `doc/prototypes/index.html` 可以在浏览器打开。
- 左侧导航可以切换所有原型屏。
- Panel 展开/折叠交互可用。
- 设置 Tab 切换可用。
- 每个原型屏都有版本来源、配置字段、验收点和降级风险说明。
- 原型引用当前橘猫素材和 app icon，不使用缺失资源。
- 原型说明文档能解释每个界面与 PRD / implementation-plan / progress 的对应关系。
- v0.4 新增屏幕覆盖默认桌宠体验、Panel 边缘状态、设置关键状态、托盘与发布包。
- v0.4 原型明确主题系统后置，不提供外观选择入口。
