# v0.9 宠物动画“玩耍优先”修订

## 状态

- 决策日期：2026-07-21
- 决策状态：项目所有者已确认
- 适用对象：Classic Pro、多多及后续遵守同一宠物包合同的角色
- 发布影响：当前带电脑的 S4/S5 动画只能作为历史过程证据，不得进入 v0.9 发布候选
- 稳定回退：Windows v0.8 默认宠物链

## 问题判断

当前素材虽然通过帧数、锚点、图集、分层和连续性门禁，但大量依赖电脑、显示器和键盘解释“工作”。结果更像状态插画，不像有生命感的桌宠；电脑的进出也让午休转换显得机械，并放大遮挡、比例和图层拼接问题。

本次不修改工资状态机，也不重命名运行时动作。修订只改变视觉叙事：状态由节奏、能量和动作选择区分，宠物本身是画面主体。

## 硬性禁用元素

以下元素不得出现在任何新生成帧、图集、Contact Sheet、GIF、运行时包或发布候选中：

- 电脑、笔记本电脑、显示器；
- 键盘、鼠标；
- 办公桌、办公椅、工位隔板；
- 模拟办公软件、文字、Logo 或屏幕界面；
- 为解释工作状态而临时加入的其他大型道具。

旧候选不删除，保留在 PetManager 历史工作区并明确标记为 `superseded_visual_direction`。

## 状态表达原则

| 基础状态 | 动作能量 | 推荐表现 | 不允许 |
|---|---|---|---|
| `working` | 中高、稳定、有节奏 | 扑抓、追尾、拍打、来回试探、短暂停顿后继续玩耍 | 坐在工位、敲键盘、盯屏幕 |
| `awake_rest` | 低到中、间歇、松弛 | 理毛、伸展、摆尾、观察、翻身、低频自娱 | 长时间完全静止或复用 working 的高频循环 |
| `sleeping` | 极低、连续、安静 | 呼吸、耳动、尾尖轻动、轻微翻身 | 完全醒来、强烈庆祝、指针跟随 |

工作状态不是“猫在办公”，而是“当前业务状态活跃时，猫更有精力地玩”。收入、工作进度和当前时段由 Panel 与今日详情表达，不由宠物道具承担。

## 动作合同

运行时 ID、动作类型、帧数和回退结构保持兼容；视觉内容按下表重制。

| 动作 ID | 新视觉语义 | 建议结构 | 道具策略 |
|---|---|---|---|
| `working_loop` | 主动玩耍循环 | 观察目标 -> 扑抓/拍打 -> 收势 -> 再次准备；首尾可无缝循环 | 首选无道具 |
| `working_ack` | 工作状态下回应用户 | 停止当前玩耍 -> 看向用户 -> 轻拍/歪头 -> 回到 active pose | 无道具 |
| `rest_ack` | 清醒休息时温和回应 | 眨眼、摆尾、抬爪或蹭头，结束后回到松弛姿势 | 无道具 |
| `sleep_ack` | 睡眠中的微回应 | 耳朵或尾尖轻动，短暂睁眼也不得完全坐起 | 无道具 |
| `run_prepare` | 长按后的跑动准备 | 压低、蓄力、朝当前方向准备起跑 | 无道具 |
| `run_stop` | 拖动释放后的收势 | 减速、刹停、站稳，再解析最新基础状态 | 无道具 |
| `lunch_relief` | 午休开始的释放感 | 停下高能玩耍 -> 大幅伸展/翻滚 -> 落到 `awake_rest` | 无道具 |
| `lunch_return` | 午休结束的重新活跃 | 从松弛姿势起身 -> 伸展 -> 进入玩耍准备姿势并衔接 `working_loop` | 无道具 |

若无道具版本在人工审查中仍不能区分状态，允许提出单一轻量玩具方案，例如小球、丝带或逗猫棒。任何玩具都必须是动作的辅助，不得遮挡身体，不得跨动作突然出现，并需项目所有者单独批准后才能生成。

## 节奏与连续性

- `working_loop` 必须比 `awake_rest` 更活跃，但不能持续快速抖动；建议单循环 1.4-2.4 秒。
- `awake_rest` 的环境动作建议间隔 20-90 秒，避免桌面持续喧闹。
- `sleeping` 以 2.5-5 秒呼吸循环为主，交互只产生低幅微动作。
- `lunch_relief` 结束帧必须可直接衔接 `awake_rest`。
- `lunch_return` 结束帧必须可直接衔接 `working_loop`。
- 同一动作内相邻帧的角色尺度、脚底线、中心点、耳朵、尾巴和四肢必须稳定。
- Classic 与多多使用同一语义合同和验证脚本，不得新增 `pet_id` 特判。

## 生产策略

1. 先为 Classic 重制 `working_loop`、`working_ack`、`lunch_relief`、`lunch_return`。
2. 锁定现有通过的 `rest_ack`、`sleep_ack`、`run_prepare`、`run_stop`，除非人工审查发现它们也不符合玩耍优先方向。
3. Classic 四项通过真实时长 GIF、Contact Sheet、边界衔接和 LMM 影子接入后，再按相同合同生成多多对应动作。
4. 旧带电脑候选不得覆盖、删除或冒充新候选；新工作区使用独立版本号和哈希。

## 人工视觉门禁

每个候选必须同时满足：

- 画面中不存在任何电脑或办公道具；
- 第一眼看到的是猫的动作，不是道具的移动；
- `working`、`awake_rest`、`sleeping` 在静音 GIF 中可辨；
- `working_loop` 连续播放 10 次不显机械、不抖动、不产生明显首尾跳帧；
- 午休开始和结束不依赖道具出现/消失表达状态；
- Classic 与多多身份、比例、颜色、耳朵、尾巴和四肢稳定；
- 无透明缺口、色键残留、相邻槽污染、基线漂移和比例跳变；
- 不修改 LMM 运行时动作 ID、输入仲裁或回退优先级。

未通过人工视觉门禁时保持 `review_pending / ready:false / published:false`。

## 与当前成果的关系

- S4/S5 对 custom profile、分层证据、图集、逐帧时长、QA 和审查门禁的工程成果继续保留并复用。
- 当前带电脑的 `working_loop`、`working_ack`、`lunch_relief`、`lunch_return` 视觉内容被产品决策替代。
- 这不是对 S4/S5 工程质量的否定，而是产品动作方向在真实观感审查后的修订。
