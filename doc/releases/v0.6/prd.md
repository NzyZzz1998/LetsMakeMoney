### 1.2 当前版本目标

v0.6 不以增加大量功能为目标，而是让 v0.5 已经可发布的桌宠能力更容易观察、复现、验证和维护，同时完成一轮有证据边界的日常体验精修。

版本成功后应达到：

1. 普通发布模式也能记录足够的关键语义事件，排查不再依赖临时开启 Debug。
2. 自动验证不会在出现阻塞错误时仍报告通过。
3. 托盘真实链路可以由仓库内标准脚本重复验证，不给正式用户暴露测试后门。
4. 用户可以打开应用数据目录并复制脱敏诊断摘要。
5. 右键菜单与托盘菜单职责清楚，Settings、Wizard、Panel 和菜单只做有证据的有限一致性修正。
6. 开机自启、损坏配置和恢复默认等低频高信任路径有明确预期、自动检查与人工验收入口。
7. 文档、原型、版本、分支和发布包口径与真实仓库状态一致。

### 1.3 已确认方案

采用推荐方案：九项需求全部在 v0.6 闭环，但区分用户功能、体验改进、工程质量、验证能力、文档治理和条件性 bugfix。

- 托盘自动验收：标准化外部 PostMessage 验证脚本，不新增内部 tray trigger。
- 轻量诊断：设置页提供“打开应用数据目录”和“复制诊断摘要”，不导出文本或 Zip。
- 诊断摘要：默认脱敏，不复制月薪、工作时间、窗口坐标、完整路径或原始日志正文。
- 体验精修：只修当前截图、日志、录屏或实际观察能够证明的问题。
- Settings / Wizard：只修共享控件接入后的残留差异，不重做结构。
- 菜单分层：桌宠右键负责上下文操作；托盘菜单负责找回和生命周期；必要的设置、关于、退出入口允许重复。
- 高信任路径：先验证，真实复测失败后才创建对应 bugfix。

### 1.4 三档方案对照

| 方案 | 覆盖范围 | 用户价值 | 工程成本 | 主要风险 | 结论 |
|---|---|---|---|---|---|
| 最小方案 | 九项只做最小闭环，不增加诊断入口，只写脚本和文档 | 发布可信度提升，但用户反馈问题仍需手工找日志 | 低 | IDEA-005 用户价值没有落地 | 仅作为回退 |
| 推荐方案 | 日志、验证、外部托盘脚本、脱敏诊断、菜单分层、有限 UI 精修、高信任路径验证 | 同时改善用户支持、发布质量和日常体验 | 中 | 需要严格控制 UI 与测试工具范围 | 本 PRD 采用 |
| 过大方案 | 诊断中心、Zip 导出、内部测试后门、验证框架重构、菜单系统重构、大型 UI 改版 | 表面完整 | 高 | 版本失控、维护成本上升、正式包暴露测试能力 | 明确不做 |

## 3. 背景、用户与核心场景

### 3.1 产品背景

LetsMakeMoney 是 Windows x86_64 桌面薪资可视化小工具，以透明无边框橘猫桌宠和暖色金币 Panel 展示今日收入、月累计、时薪和工作进度。v0.5 已完成 Settings / Wizard 共享控件、托盘恢复、纯桌宠任务栏策略、点击穿透保护和保存反馈收敛。

v0.5 验收暴露的主要问题不是核心功能缺失，而是：

- `verify_v05.ps1` 在 Godot 输出 Parser/Error 后仍可能以成功退出。
- 点击穿透暂停/恢复等关键事件在普通模式不可见。
- 托盘自动验收依赖临时同路径补证，仓库内没有稳定工具。
- 用户反馈问题时需要手工寻找 `config.json` 与 `debug.log`。
- 普通单双击会持续生成诊断截图，形成不必要的本地存储增长。
- README、原型和 About 等位置仍残留旧版本口径。
- 开机自启、损坏配置和恢复默认缺少同一套可信验收标准。

### 3.2 目标用户

- 日常使用橘猫桌宠查看收入进度的 Windows 用户。
- 需要重新配置薪资、显示方式、托盘和开机启动的用户。
- 收集用户问题、复现故障、打包和发布 Beta 版本的项目维护者或 agent。

### 3.3 核心场景

1. 用户正常运行桌宠，出现托盘、窗口或点击穿透问题后，可以快速提供脱敏诊断摘要。
2. 维护者运行验证脚本，看到“通过”时能够相信不存在阻塞解析或运行错误。
3. 发布前自动验证托盘隐藏、恢复和纯桌宠任务栏策略，并保留人工真实托盘补测入口。
4. 用户配置损坏或恢复默认失败时，应用不会静默丢失原配置，也不会显示虚假的成功状态。
5. 用户打开桌宠菜单、Settings 或 Wizard 时，视觉层级保持一致，但不会因为 v0.6 再次发生大规模 UI 返工。

## 5. 功能需求

### FR-001 默认日志口径与本地诊断产物治理

- 上游：IDEA-001
- 类型：工程质量 / 验证能力
- 入口：应用启动、配置读写、Settings/Wizard、Popup、托盘、窗口策略、点击穿透和诊断操作。
- 前置条件：应用可以访问 `%APPDATA%\LetsMakeMoney`；无法访问时应用仍应继续运行并把错误输出到 Godot 标准错误通道。

#### 正常流程

1. 应用按 `info`、`debug`、`error` 三类写入 `debug.log`。
2. 普通模式默认写入低频、可验收的语义事件。
3. `debug_mode=true` 时额外记录坐标、命中区、缓存、帧和 native 细节。
4. 写入前检查日志大小；达到 2 MB 时将当前文件轮换为 `debug.log.1`。
5. 只保留当前日志和一个轮换备份；新轮换覆盖旧的 `.1`。
6. 单击/双击交互截图只在 `debug_mode=true` 或隔离验证环境中生成。

#### 默认 info 事件白名单

- 应用启动、版本、native health 和退出。
- 配置不存在、加载成功、加载失败、默认恢复、保存成功和保存失败。
- Settings 打开/关闭及保存成功、无变化、失败。
- Wizard 打开、步骤切换、完成、失败、取消和关闭。
- Modal/Popup 打开与关闭。
- `passthrough_suspended`、`passthrough_resumed` 和恢复失败。
- 托盘左键、托盘菜单命令、窗口显隐结果。
- 纯桌宠、任务栏可见性和窗口策略重应用结果。
- 开机自启、恢复默认和诊断操作结果。

#### 分支与异常

- 日志目录无法创建：不阻塞应用启动；输出可读错误，但不循环重试刷屏。
- 日志轮换失败：继续尝试追加当前文件；记录一次错误。
- Debug 关闭：不生成交互截图，不写入坐标、命中区或逐帧日志。
- 现有 `interaction-screenshots` 文件不自动删除，避免未经用户确认删除本地数据。

#### 状态与数据

- 不新增配置字段；沿用 `debug_mode`。
- 日志路径：`%APPDATA%\LetsMakeMoney\debug.log`、`debug.log.1`。
- 交互截图路径保持现状，但生成条件改变。

#### 旧体验保护

- 不改变桌宠点击、双击、长按、拖拽和右键行为。
- 不因日志写入失败阻塞桌宠主流程。

#### 影响范围

- **Godot 运行时**：统一日志等级、语义事件写入和日志轮换；调整普通模式与 Debug 模式的截图门控。
- **本地文件**：涉及 `%APPDATA%\LetsMakeMoney\debug.log`、`debug.log.1` 和现有 `interaction-screenshots` 目录；不自动删除历史截图。
- **配置**：只读取现有 `debug_mode`，不新增配置字段，也不改变其用户语义。
- **交互模块**：Settings、Wizard、Popup、托盘、窗口策略和点击穿透只补语义日志，不改变既有流程。
- **验证与文档**：验证脚本需检查日志等级、轮换和截图门控；手动验证文档需记录普通模式与 Debug 模式差异。

#### 验收

- 普通模式完成 Settings、Wizard、托盘和 Popup 操作后，关键 info 事件完整。
- 普通模式连续单双击各 10 次，不新增交互截图。
- Debug 模式触发点击时仍可按需生成截图。
- 在隔离目录构造超过 2 MB 的日志后，轮换结果只包含 `debug.log` 与 `debug.log.1`。

### FR-002 活跃验证脚本输出净化与可信退出

- 上游：IDEA-002
- 类型：工程质量
- 入口：仓库根目录运行 v0.6、v0.5、v0.4 活跃验证和文档检查命令。
- 出口：明确的成功摘要或非零失败退出。

#### 正常流程

1. 验证使用隔离的 `APPDATA` / `LOCALAPPDATA`，不污染用户配置和日志。
2. 脚本捕获 Godot 标准输出、标准错误和退出码。
3. 验证脚本先完成必要的类缓存/资源导入准备，再运行验证主体。
4. 只有退出码为 0 且没有阻塞错误文本时才输出一次通过。
5. 失败时显示首个阻塞错误、完整日志文件路径和非零退出码。

#### 阻塞错误

- `Parser Error` / `Parse Error`
- `Script Error`
- `Failed to load script`
- `Failed to instantiate an autoload`
- `Invalid call`
- `Node not found`
- `null instance`
- 未处理异常或进程提前退出

Godot 引擎版本标题和已明确审核的 warning 可以保留，但必须通过小型 allowlist 管理，不能用大范围文本过滤隐藏未知错误。

#### 影响范围

- **PowerShell 验证入口**：新增 v0.6 主验证入口，并修正当前 `verify_v05.ps1` 在阻塞错误存在时仍可能报告通过的问题。
- **共享脚本 helper**：如能降低重复，可让 v0.4/v0.5/v0.6 活跃脚本复用错误扫描、隔离目录和退出码处理；不迁移 v0.1-v0.3 历史脚本。
- **Godot 验证脚本**：不改变业务行为，只允许为稳定加载顺序、明确测试契约和输出摘要做必要调整。
- **用户本地数据**：验证过程使用隔离 `APPDATA` / `LOCALAPPDATA`，不得读写真实配置、日志、截图或注册表状态。
- **文档与发布门禁**：verification、release checklist 和 CI/本地命令必须以非零退出作为阻塞失败依据，不能只检查“通过”文本。

#### 验收

- 注入已知 Parser Error 时脚本必须非零退出。
- 正常项目验证只出现一次通过摘要。
- 失败输出必须指向可读取的日志文件。
- 运行后用户真实 `%APPDATA%\LetsMakeMoney` 内容不变化。

### FR-003 外部托盘真实链路验收工具

- 上游：IDEA-003
- 类型：验证能力 / Windows native
- 入口：PowerShell 运行仓库内托盘专项验证脚本。
- 前置条件：Windows x86_64、v0.6 发布候选 exe 和 native DLL 可用。

#### 正常流程

1. 脚本备份用户当前配置，并在隔离或可恢复状态下启动发布候选 exe。
2. 脚本查找 `LetsMakeMoneyTrayMessageWindow` 原生隐藏消息窗口。
3. 通过 `PostMessage` 向 `TRAY_CALLBACK_MESSAGE` 发送与托盘左键相同的 Windows 消息。
4. 应用经过 `TrayController.window_proc`、`poll_tray_command`、`Platform` signal 和 `Main` 显隐处理链路。
5. 脚本分别验证 `pure_pet_mode=false` 和 `true`：
   - 两种模式均可隐藏并恢复窗口。
   - 非纯桌宠恢复后存在任务栏入口。
   - 纯桌宠恢复后不出现任务栏入口。
6. 脚本检查 `debug.log` 的托盘、窗口策略和任务栏语义事件。
7. 无论成功或失败，均恢复原配置并结束测试进程。

#### 边界

- 不新增 Godot UI、配置字段或 native 内部 trigger。
- 验证脚本不打入正式发布 Zip。
- 外部 PostMessage 覆盖原生回调之后的真实应用链路，但不能完全替代 Windows Explorer 通知区的人工点击，因此发布前仍保留一次人工真实托盘补测。

#### 异常流程

- 找不到原生消息窗口：失败并输出 native 初始化日志。
- 应用提前退出：失败并输出进程退出码。
- 显隐或任务栏策略不符：记录为 bugfix，不通过增加测试绕过。
- 配置恢复失败：脚本必须报告并保留备份路径。

#### 影响范围

- **PowerShell 验证脚本**：新增外部托盘专项脚本，负责启动、PostMessage 触发、日志检查、进程结束和状态恢复。

### FR-005 轻量诊断与支持

- 上游：IDEA-005

```text
[打开应用数据目录]  [复制诊断摘要]

```

该区域使用 Settings 现有行式布局和 Warm Control，不增加新页面、右键菜单项或托盘菜单项。

#### 打开应用数据目录

1. 确认 `%APPDATA%\LetsMakeMoney` 存在；不存在时先创建。
2. 使用系统文件管理器打开该目录。
3. 成功显示轻量反馈并记录日志。
4. 失败保留 Settings，显示可读错误，不弹系统阻塞对话框。

#### 复制诊断摘要

摘要允许包含：

- 应用版本、构建类型、Windows 平台和时间。
- `config_version`。
- 当前 `pet_id`。
- `debug_mode`、`window_mode`、`pure_pet_mode`、托盘、穿透和透明窗口等非敏感开关。
- native health 各能力布尔值和最近错误。
- 日志文件是否存在、大小、最后更新时间。
- 最近关键语义事件是否出现及最后时间，不复制原始日志正文。
- 交互截图目录是否存在和占用大小。

最近语义事件按以下规则读取：

1. 先读取 `debug.log`，再读取 `debug.log.1`。
2. 每个文件只从尾部读取实现所需的有界内容，不把完整日志载入摘要或剪贴板。
3. 同一事件只输出最后出现时间和是否存在，不复制原始日志行。

摘要禁止包含：

- 月薪、工作时间和每日工作小时数。
- 窗口坐标。
- 用户名、完整本地路径或计算机名。
- 原始 `config.json`。
- 原始日志正文或交互截图内容。

#### 状态

- 成功：`诊断摘要已复制。`
- 无变化不适用；重复点击仍重新生成当前摘要。
- 暂无日志：仍复制基础摘要，并标记“日志不可用”。
- 剪贴板不可用：显示失败，摘要不写入其他文件。
- native 不可用：摘要正常生成并标注能力不可用。
- 初始状态不显示反馈区域，也不保留“等待操作”占位。
- 操作反馈显示 2 至 3 秒后自动隐藏；重复操作重新计时。
- 若启动时发生损坏配置恢复，Settings 通用页首次打开时显示一次“配置已恢复为默认值，原文件已保留”的非阻塞提示；提示只在当前进程内消费，不新增配置字段。

#### 剪贴板成功判据

1. 先在内存生成脱敏摘要并完成字段白名单检查。
2. 调用系统剪贴板写入后，读取剪贴板内容并与本次摘要做完整比对。
3. 只有读回内容完全一致时记录 `diagnostics_copy_success` 并显示成功。
4. 写入、读回或比对任一步不可用或不一致时记录 `diagnostics_copy_failed`，显示失败且不得伪报成功。
5. 失败时不把摘要写入其他文件，也不改变 Settings 表单和配置。

#### 数据与配置

- 不新增配置字段。
- 应用版本从项目统一版本元数据读取，不在 UI 脚本中硬编码多个版本。
- 不导出文本或 Zip，不上传任何数据。

#### 影响范围

- **Settings UI**：仅在“通用”页增加“诊断与支持”区域、两个动作按钮和轻量反馈，不新增页签或重排现有信息架构。
- **Godot 能力**：新增打开应用数据目录、生成脱敏摘要和写入剪贴板的本地调用；失败不得阻塞 Settings。
- **数据读取**：只读统一版本元数据、非敏感配置、native health、日志元数据和截图目录占用，不复制原始配置或日志正文。
- **配置与持久化**：不新增配置字段，不导出 txt/Zip，不上传数据，也不在磁盘保存诊断摘要。
- **日志、原型与验证**：新增诊断成功/失败语义事件；原型覆盖成功、日志缺失、失败和禁用状态；验证脚本检查脱敏白名单。

#### 验收

- 摘要不含薪资、时间、坐标、用户名和绝对路径。
- 日志缺失、native 缺失和剪贴板失败均有可读反馈。
- 操作反馈 2 至 3 秒后隐藏，不常驻占用大块空间。

### FR-009 开机自启、配置恢复和恢复默认专项验证

- 上游：IDEA-009
- 类型：验证能力 / 条件性 bugfix
- 原则：先建立可重复验证；只有真实结果不符合以下预期时才进入 bugfix。

#### 开机自启预期

1. 仅导出的 `LetsMakeMoney.exe` 支持开机自启。
2. 开启后 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run\LetsMakeMoney` 指向当前 exe 的带引号绝对路径。
3. 关闭后对应注册表值不存在。
4. 设置失败时 UI 与配置都反映真实状态，不能显示成功或保存错误布尔值。
5. 自动验证必须备份并恢复原注册表值，不留下测试状态。

#### 配置加载与恢复预期

- 配置不存在：加载默认值，并按现有首次启动规则进入 Wizard。
- 部分旧配置：通过 defaults merge 补齐缺失字段，已有合法值保留。
- JSON 损坏：将原文件保留为带时间戳的 `.invalid.json`，加载默认值，记录 `config_load_failed` 和 `config_recovered_defaults`。
- 损坏恢复不弹阻塞式启动对话框；在 Settings 通用页诊断区域和诊断摘要中显示一次非阻塞提示。
- 配置不可写：保存失败，保留用户输入，内存配置回滚，日志记录可读原因。

#### 恢复默认显示设置预期

- 只恢复窗口位置、窗口模式、Debug、自启动、托盘、native、穿透、透明窗口、纯桌宠、透明度和缩放。
- 保留月薪、休息模式、工作时间、宠物选择和 `panel_items`。
- 关闭系统自启动失败时，不得把 `auto_start` 配置伪装成 `false`。
- 保存失败时恢复内存快照，并显示失败反馈。

#### 配置与外部副作用事务

Settings 普通保存和恢复默认共用以下事务边界：

1. 记录 Config 数据快照、当前宠物 ID、当前开机启动注册表值与窗口运行态。
2. 在改变 Config 前先验证目标宠物、目标字段和外部能力是否可用。
3. 开机启动变更失败时立即中止，不修改 Config 或宠物运行态。
4. 配置持久化使用临时文件写入、读回校验和原子替换，或具备同等“失败不破坏旧配置”的实现。
5. 配置保存失败时恢复原注册表值、Config 内存快照、宠物运行态和窗口策略；用户表单输入继续保留。
6. 配置保存成功后再应用宠物切换和窗口策略；运行态应用失败必须记录错误并恢复到保存前可用状态。
7. 恢复默认关闭自启动成功、但后续配置保存失败时，必须重新写回原注册表值；补偿失败作为发布阻塞错误记录。

#### 影响范围

- **Settings 通用页**：覆盖开机自启、恢复默认显示设置、失败反馈和配置恢复提示；验证通过时不预设 UI 重做。
- **Config**：覆盖配置不存在、旧配置缺字段、损坏 JSON、不可写和默认恢复；新增的仅是损坏文件时间戳备份，不新增业务字段。
- **Windows 注册表**：读写并恢复 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run\LetsMakeMoney`，自动测试不得污染用户原值。
- **运行时状态**：验证恢复默认后窗口、托盘、native、穿透、纯桌宠、透明度和缩放与配置一致，同时保留薪资、时间、宠物和 Panel 项。
- **验证脚本、日志与文档**：新增配置/自启动专项验证和可恢复夹具；只有真实失败才进入 bugfix，并在验证文档记录失败证据。

#### 验证矩阵

| 场景 | 次数 | 自动检查 | 人工检查 |
|---|---:|---|---|
| 自启动开启/关闭 | 各 3 次 | 注册表路径、返回值、配置一致 | Settings 反馈 |
| 配置不存在 | 2 次 | 默认值和 Wizard 入口 | 首次启动体验 |
| 部分旧配置 | 3 个样本 | defaults merge | Settings 显示 |
| 损坏 JSON | 3 个样本 | invalid 备份、默认恢复、日志 | 非阻塞提示 |
| 恢复默认 | 3 次 | 保留字段、重置字段、注册表 | 反馈与运行状态 |
| 配置不可写 | 1 次 | 回滚和错误日志 | 输入保留、失败文案 |

- 桌宠右键菜单及两个二级菜单。

### 9.5 通过标准

- P0 项全部通过；任何 Parser/Script/Invalid call 阻塞错误均为失败。
- 托盘脚本两种模式各 10 轮零失败，人工真实托盘路径至少 1 轮通过。
- 普通模式 20 次点击不新增交互截图。
- 诊断摘要 5 次抽检均无薪资、工作时间、坐标、用户名、绝对路径或原始日志。
- 损坏配置 3 个样本均保留 invalid 备份并安全恢复。
- 恢复默认不改变薪资、宠物和 Panel 项。
- Settings/Wizard/Panel/菜单不存在文字裁切、默认深色 popup 或交互回归。
- Settings 取消/关闭零副作用；保存失败后 Config、宠物运行态和注册表均恢复，表单输入保留。
- Wizard 取消/关闭/完成失败不留下未保存宠物或配置变化。
- 原生托盘菜单不要求 Godot Popup 穿透日志；桌宠右键菜单和 Modal 的暂停/恢复日志必须成对出现。
- 文档扫描不再出现已知 v0.4/test/待打 v0.5 tag 旧口径。

### 9.6 不通过处理

- 工程脚本失败：修复脚本或加载顺序，不允许扩大 allowlist 掩盖未知错误。
- 托盘/native 失败：记录 v0.6 bugfix，保留人工找回路径；不增加正式包测试后门。
- 配置/自启动失败：记录为发布阻塞 bugfix，确保 UI 与真实系统状态一致。
- UI 证据不充分：标记“已审查、待验证”，不制造无证据改动。
- 诊断摘要泄露敏感字段：立即阻塞发布并回退诊断复制能力。
- 旧体验回归：优先回退对应模块的 v0.6 修改，不改动 v0.5 已验证业务语义。

### 9.7 手动验证文档

必须创建 `doc/releases/v0.6/verification.md`，使用可填写“结果、备注、截图/日志证据”的统一格式，至少覆盖：

- 诊断与隐私。
- Panel/菜单/Settings/Wizard 视觉对照。
- 自启动、配置恢复、恢复默认。
- v0.5 核心交互回归。

## 10. 风险与回退

| 风险 | 影响 | 控制方式 | 回退 |
|---|---|---|---|
| info 日志过多 | 日志增长、噪音 | 语义白名单 + 2 MB 轮换 | 将高频事件降回 debug |

| 自启动测试污染注册表 | 用户启动行为改变 | 备份/恢复、测试后复查 | 恢复原注册表值 |
